## 2. 평가 방법론

### 2-1. 학술적 근거

본 평가는 아래 프레임워크를 참조하여 설계되었다:

| 프레임워크 | 출처 | 적용 |
|-----------|------|------|
| **RAGAS** (Shahul Es et al., 2023; v0.2 2024) | Retrieval Augmented Generation Assessment | faithfulness, answer_correctness 개념 차용 |
| **LLM-as-a-Judge** (Zheng et al., NeurIPS 2023) | MT-Bench / Chatbot Arena | 5차원 비교 루브릭 설계 |
| **FActScore** (Min et al., 2023) | Atomic Fact 분해 + 검증 | 할루시네이션 정성 분석 |
| **SAFE** (Google DeepMind, 2024) | Search-Augmented Factuality Evaluator | "정직한 거부" 가치 분석 |
| **Seven Failure Points of RAG** (Barnett et al., 2024) | RAG 실패 유형 분류 | 근본 원인 분석 |

### 2-2. 평가 차원 (5차원 루브릭)

| # | 차원 | 설명 | RAG 차별화 |
|---|------|------|-----------|
| 1 | **사실 정확성** | 규제·세금·대출 사실의 정확도 | document pass 시 우위 |
| 2 | **근거 제시** | 출처/법적 근거 인용 여부 및 품질 | RAG 고유 강점 |
| 3 | **완전성** | 질문의 모든 측면을 다루는 정도 | Opus 우위 |
| 4 | **실용성** | 사용자가 행동할 수 있는 구체적 조언 | Opus 우위 |
| 5 | **안전성** | 불확실 시 정직 거부, 면책, 시간 인식 | RAG 우위 |

### 2-3. 분석 방법

- **정량 분석**: `result/summary.md` 및 `opus_result/summary.md`의 통계 데이터
- **카테고리별 비교**: Part A 7개 카테고리(A-1~A-7) + Part C + Part D
- **정성 비교**: 카테고리별 대표 질문의 실제 답변 비교 (세금 15개, 전세 15개, 매매 15개, 대화형 20개 심층 분석)
- **바이어스 인지**: LLM-as-Judge 바이어스(Position, Verbosity, Self-Enhancement, Authority) 문헌 기반 고려

## 3. 시스템 개요 비교

### 3-1. RAG+LLM 시스템 (Sonnet)

```
사용자 질의
    │
    ▼
[QueryAnalyzer] — SIMPLE(80%) / REWRITE(16%) / DECOMPOSE(4%)
    │
    ▼
[BGE-M3 Dense+Sparse RRF 검색] — domain_ontology_v2 + legal_docs_v2
    │                               (realestate_v2 미연동)
    ▼
[Cross-Encoder bge-reranker-v2-m3-ko]
    │
    ▼
[CRAG Compensator] — CORRECT / AMBIGUOUS / INCORRECT
    │
    ▼
[grade_documents] — pass/fail 판정
    │
    ├── pass → [generate] → [check_hallucination] → 답변
    │
    └── fail → [rewrite_query] → 재검색 → [generate] → [check_hallucination]
```

- **검색 범위**: domain_ontology_v2(2,146 포인트) + legal_docs_v2(976 포인트) = 3,122 포인트
- **미연동**: realestate_v2(93,943 포인트, YouTube 전문가 콘텐츠) — **핵심 코퍼스 제외**
- **LLM 호출**: 질문당 평균 3.88회 (분류 + CRAG + 문서평가 + 재작성 + 생성 + 할루시네이션 검증)

### 3-2. 순수 LLM 시스템 (Opus 4.6)

```
사용자 질의
    │
    ▼
[Claude Opus 4.6] — 파라메트릭 지식만 사용
    │
    ▼
답변 (+ 41% 확률로 후속 제안)
```

- **시스템 프롬프트**: "당신은 대한민국 부동산 전문 상담사입니다. 한국어로 답변하세요. 구체적이고 실용적으로 답변하세요."
- **LLM 호출**: 질문당 1회 (단순 프롬프트 → 응답)
- **후속 제안**: Opus가 41%의 질문에서 자발적으로 후속 질문 제안, 99.7% 수락

### 3-3. 비용/성능 비교

| 항목 | RAG (Sonnet) | Opus 4.6 | 비율 |
|------|-------------|----------|------|
| 질문당 비용 (CLI) | $0.032 | $0.075 | RAG 2.3× 저렴 |
| 질문당 시간 | 91초 | 36초 | Opus 2.5× 빠름 |
| API 전환 시 예상 시간 | 5-8초 | 동일 | — |
| 평균 출력 토큰 | ~600 | ~1,288-2,730 | Opus 2-4× 길음 |
| LLM 호출 수 | 3.88회 | 1회 | RAG 3.88× 많음 |

> RAG의 91초 레이턴시는 CLI subprocess 방식 때문이며, API 전환 시 5-8초로 단축 예상.

## 5. 카테고리별 상세 분석

### 5-1. 세금 (A-2, Q16-30) — RAG 최강 카테고리

**문서평가 pass 73%** — RAG가 가장 경쟁력 있는 영역이다. legal_docs_v2에 "주택과 세금" 법률 문서가 포함되어 있어, 세금 관련 질문에서 법적 근거를 직접 인용할 수 있다.

#### RAG 출처 인용의 실질적 가치 — Q16, Q27

**Q16: "취득세 얼마 나오나요? 9억짜리 아파트 1주택자요"**

RAG(doc pass, halluc fail):
- 법률 문서 기반으로 3% 세율 계산: 27,000,000원(취득세) + 2,700,000원(지방교육세) = **29,700,000원**
- 출처: `[8] 법률: 제1편 주택의 취득과 관련된 세금 — legal_docs_v2`
- 사용자가 법적 근거를 **직접 추적 가능**

Opus:
- 동일한 3% 세율과 계산 결과 제공
- 추가로 조정대상지역 2주택 8%, 3주택+ 12% 중과 정보 포함
- **출처 미인용** — 수치의 근거를 확인할 방법 없음

→ **RAG의 가치**: 동일한 사실이라도 "법률 제X편에 기반" vs "그냥 그렇습니다"의 차이. 세금 상담에서 출처 추적 가능성은 실질적 가치.

**Q27: "재산세 6월 1일 기준이라 잔금일 조절해야 한다면서요?"**

RAG(doc pass, halluc pass):
> "잔금일과 등기일이 다를 경우, 둘 중 **더 빠른 날**이 취득일입니다" [7]
> "6월 2일 이후 잔금 → 당해 연도 재산세 부담 없음"

- 이 "빠른 날" 규칙은 **비직관적**이며 문서에서만 정확히 확인 가능한 인사이트
- Opus도 같은 내용을 다루지만 법적 근거 없이 일반 지식으로 제시

→ **RAG의 고유 가치**: 비직관적 법적 규칙을 문서 기반으로 정확히 전달.

#### 발견된 문제 — Q17 문서 시점 불일치

**Q17: "양도세 비과세 조건"**

| 항목 | RAG | Opus |
|------|-----|------|
| 비과세 기준 | **8억원 이하** | **12억원 이하** |
| 근거 | legal_docs_v2 인용 | "2021.12.8 이후 양도분부터 9억 → 12억 상향" |

RAG가 인용한 법률 문서가 **2021년 12월 개정 이전 기준**을 포함하고 있을 가능성이 높다. 이는 RAG 시스템의 구조적 위험을 보여준다: **출처가 있어도 그 출처가 최신이 아닐 수 있다.**

→ **개선 필요**: 문서 upload_date 기반 최신성 검증 + "이 정보는 {날짜} 기준" 면책 표시.

#### RAG의 정직한 거부 가치 — Q21

**Q21: "증여 vs 상속 뭐가 절세에 유리해요?"**

RAG(doc fail, halluc pass):
> "증여세율 vs 상속세율 구체적 비교...제공된 자료에서 확인할 수 없습니다."
> "다만 동거주택 상속공제 최대 6억원은 확인됩니다." [출처]

- 문서 범위 밖임을 **명시적으로 고지**
- 확인 가능한 부분만 선별적으로 답변
- Opus는 전체 비교표를 자신감 있게 제공하나, 수치 검증 불가

→ RAG의 부분 답변 + 정직 거부가 "틀린 확신"보다 사용자에게 안전.

### 5-2. 전세/임대차 (A-4, Q46-60) — RAG 중간 강점

**문서평가 pass 47%** — 세금 다음으로 RAG가 경쟁력 있는 영역. 주택임대차보호법 관련 문서가 legal_docs_v2에 포함되어 있다.

#### RAG 법적 구조 설명의 강점 — Q52, Q53

**Q52: "확정일자 없으면 보증금 못돌려받나요?"**

RAG(doc pass, halluc pass):
- 3중 보호 메커니즘을 법적 구조에 따라 분리 설명:
  1. **우선변제권** (확정일자 필요) — [1] 인용
  2. **소액임차인 최우선변제** (확정일자 없어도 적용) — [5] 인용
  3. **전세보증보험** (별도 가입 시) — [3] 인용
- 비용 600원 명시, 즉시 받으라는 실용 조언 포함

Opus:
- 동일 3중 구조 + 2026년 서울 기준 "보증금 1억 7,500만원 이하 → 최대 5,833만원 최우선변제" 수치 제공
- 더 상세하나 출처 미인용

→ RAG가 법적 "왜(Why)" 구조를 정확히 전달하는 반면, Opus는 "무엇(What)" 수치를 더 풍부하게 제공.

**Q53: "경매로 넘어간 전세집 보증금 받을 수 있나요?"**

RAG(doc pass, halluc pass):
- 세 가지 회수 경로 (우선변제 배당 / 소액임차인 / 전세권 등기)를 구조적으로 분리
- "지역별로 소액보증금 기준이 다르며...해당 기준 이하여야 보호받습니다" [4] — **정확한 주의**

Opus:
- "서울: 1억 6,500만원 이하 → 5,500만원까지" — 구체적 수치
- 배당요구 신청 방법, 배당순위 등 절차적 디테일 풍부

→ RAG의 "지역별 상이" 주의가 Opus의 특정 수치보다 **더 정확**한 경우. 수치는 매년 변경되므로 false precision 위험.

#### 전세 카테고리 한계 — Q48, Q50

**Q48: "전세보증보험 HUG랑 SGI 차이?"** → RAG doc fail. 보험 상품 비교 데이터 부재.
**Q50: "계약갱신청구권 2년 더?"** → RAG doc fail. 법률명은 인식하나 세부 조항 부재.

→ legal_docs_v2가 "주택과 세금" 1건에 한정되어, 임대차보호법 전문 미포함.

### 5-3. 매매/시세 (A-1, Q1-15) — Opus 압도적 우위

**RAG doc pass 50%이나 실질 답변 가치 낮음** — 시세·시황 데이터가 DB에 없어, RAG는 "확인할 수 없습니다"로 정직 거부하거나 세금/규제 측면만 부분 답변.

#### 대표 비교 — Q1

**Q1: "지금 서울 아파트 사야 할까요 아니면 더 기다려야 할까요?"**

| 차원 | RAG | Opus |
|------|-----|------|
| 사실 정확성 | — (답변 거부) | 3/5 (일반 프레임워크) |
| 근거 제시 | 5/5 (문서 한계 명시) | 1/5 (출처 없음) |

## 6. 할루시네이션 분석

### 6-1. RAG 할루시네이션 통제: 84.3% pass의 의미

| 조합 | 건수 | 비율 | 해석 |
|------|------|------|------|
| doc pass + halluc pass | ~20 | ~10% | 이상적: 충분한 문서 + 정확한 답변 |
| doc fail + halluc pass | ~146 | ~74% | "확인할 수 없습니다" → 할루시네이션 방지 |
| doc pass + halluc fail | ~1 | ~0.5% | 문서 있으나 LLM이 추가 정보 삽입 |
| doc fail + halluc fail | ~30 | ~15% | 문서 부족 + LLM 일반 지식 추가 |

**핵심 인사이트**: RAG의 높은 할루시네이션 pass율(84.3%)은 **답변 품질이 높아서가 아니라, 대부분 "모르겠다"고 거부하기 때문**이다. 이는 SAFE(Google DeepMind, 2024)의 관점에서 "정직한 거부는 할루시네이션보다 사용자에게 덜 해롭다"는 원칙과 일치한다.

### 6-2. Opus의 "자신감 있는 부정확" 위험

Opus는 197개 질문에 대해 한 번도 "모르겠다"고 하지 않는다. 이는 사용자 만족도를 높이지만, 검증되지 않은 수치를 자신감 있게 제시하는 위험을 수반한다.

#### 확인된 사례: Q17 양도세 비과세 기준

- Opus: "12억원 이하" (2021.12.8 이후 개정 기준)
- RAG: "8억원 이하" (개정 이전 기준)

두 시스템 모두 자신감 있게 수치를 제시하지만, **하나는 틀렸다**. RAG는 문서 기반이나 문서가 오래되었고, Opus는 파라메트릭 지식이나 학습 시점에 의존한다.

→ 세금/법률 영역에서는 **두 시스템 모두** 시점 검증이 필요하며, RAG는 문서 날짜 표시, Opus는 "학습 데이터 기준" 면책이 필수.

### 6-3. Barnett et al.(2024) 실패 유형 기준 분류

| 실패 유형 | RAG에서의 발현 | Opus에서의 발현 |
|----------|-------------|-------------|
| **Partial Retrieval** | 89.3% doc fail → 정직 거부 | N/A (검색 없음) |
| **Outdated Grounding** | Q17 세금 기준 시점 문제 | 학습 데이터 cutoff 의존 |
| **Extrinsic Hallucination** | 15.7% (doc fail에서 일반 지식 추가) | 검증 불가 (전체 답변이 파라메트릭) |
| **Context-Conflation** | 낮음 (검색 결과 적어서) | N/A |
| **Intrinsic Hallucination** | 매우 낮음 | N/A |

### 6-4. "정직한 거부" vs "부정확한 답변" 가치 비교

SAFE(2024) 프레임워크를 적용한 안전성 평가:

| 시나리오 | RAG 행동 | Opus 행동 | 사용자 위험 |
|---------|---------|---------|-----------|
| 문서 충분 + 최신 | 정확한 인용 답변 | 정확한 일반 답변 | **낮음** (둘 다 안전) |
| 문서 부재 | "확인할 수 없습니다" | 자신감 있는 답변 | **Opus 위험** (검증 불가) |
| 문서 있으나 오래됨 | 오래된 수치 인용 | 가능한 최신 수치 | **RAG 위험** (문서 신뢰 오판) |
| 주관적 판단 필요 | 거부 | 포괄적 프레임워크 | **RAG 과소 제공** |

→ RAG는 "모르는 것"에 안전하고, Opus는 "아는 것"에 효과적. 이상적 시스템은 **RAG의 문서 근거 + Opus의 파라메트릭 보완을 결합**해야 한다.

### 7-1. 정규 vs 구어체 격차: 보편적 패턴

→ 우리 시스템의 정규-구어체 격차(24%p)는 업계 상한에 위치하나, RAG 시스템의 **보편적 한계** 내에 있다.

### 7-2. Harvey AI 패턴과의 비교

Harvey AI(법률 RAG, 2025):
- 전략: **Recall over Precision** — 넓은 검색 후 LLM이 노이즈 필터링
- 환각률: ~0.2%
- Am Law 100 도입률: 97%

**우리 시스템과의 차이**: Harvey AI는 대규모 법률 코퍼스 전체를 검색하여 recall을 극대화하고 LLM이 precision을 보장한다. 우리 RAG는 **3,122 포인트의 극소량 DB**에서 precision을 추구하여, recall 자체가 근본적으로 부족하다.

→ **Harvey AI 전략 채택 시사점**: realestate_v2(93,943 청크) 연동으로 recall을 먼저 확보하고, 기존 CRAG + hallucination check로 precision을 보장하는 것이 합리적.

### 7-3. 한국어 특수성

TWICE(ICLR 2025) 연구:
- 한국어 네이티브 벤치마크에서 번역 대비 **5-8% 성능 저하** (다국어 모델)
- KURE-v1(한국어 파인튜닝)이 BGE-M3(다국어)보다 안정적

→ Phase 2에서 BGE-M3를 채택했으나, 한국어 5-8% 핸디캡이 존재할 수 있음. Dense+Sparse+ColBERT 3중 벡터의 유연성으로 보상.

Part D에서 RAG의 구조적 실패의 원인:
- 현재: 각 턴을 **독립 질의**로 처리
- 필요: 대화 히스토리 기반 **코레퍼런스 해소** + 컨텍스트 누적

`00_overview.md`에서 이미 설계된 코레퍼런스 해소 메커니즘 활용:
```yaml
conversation:
  max_turns: 5
  coreference_markers: ["그거", "이거", "그건", "그럼", "거기"]
```

### 8-6. P4: 문서 시점 검증 체계

Q17 사례(8억 vs 12억)에서 드러난 문제:
- 문서가 존재해도 **최신이 아닐 수 있음**
- 사용자는 RAG의 출처 인용을 "최신 정보"로 신뢰할 가능성

해결:
1. 답변 시 문서의 `upload_date` 자동 표시: "이 정보는 {날짜} 기준입니다"
2. 2년 이상 경과 문서에 경고: "⚠ 이 정보는 {년}년 기준이며, 현행 법령과 다를 수 있습니다"
3. 동일 주제 최신 문서 우선 랭킹

### 8-7. 우선순위 종합

| 순위 | 작업 | 예상 효과 | 소요 | ROI |
|------|------|---------|------|-----|
| **P0** | realestate_v2 연동 | doc pass 10% → 40%+ | 1-2일 | **최고** |
| **P1** | fallback 검색 전략 | 나머지 fail 절반 추가 커버 | 2-3일 | 높음 |
| **P2** | 하이브리드 답변 | Opus급 완전성 확보 | 3-5일 | 중간 |
| **P3** | 멀티턴 컨텍스트 | Part D 구조적 해결 | 2-3일 | 높음 |
| **P4** | 문서 시점 검증 | Q17류 오류 방지 | 1일 | 중간 |

## 9. 결론

### RAG의 현재 위치

RAG+LLM 시스템은 **"세금/법률 전문 상담사"로서는 경쟁력이 있으나, "범용 부동산 어드바이저"로서는 구조적으로 불완전**하다.

| 역할 | RAG 적합도 | 근거 |
|------|-----------|------|
| 세금 상담 | ★★★★☆ | doc pass 73%, 법적 근거 인용 |
| 임대차 법률 | ★★★☆☆ | doc pass 47%, 법적 구조 설명 |
| 시장 분석 | ★☆☆☆☆ | 데이터 부재, 정직 거부 |
| 생활 상담 | ★☆☆☆☆ | 코퍼스 범위 밖 |
| 대화형 상담 | ★☆☆☆☆ | 멀티턴 부재 |

### 향후 경로

**realestate_v2 연동(P0)만으로도 시스템의 성격이 근본적으로 변화**할 수 있다:
- 3,122 → 97,065 검색 포인트 (31× 확대)
- "세금 전문가" → "부동산 종합 어드바이저"로 역할 확장
- 이후 P1-P4를 순차 적용하여 Opus 수준의 답변 품질을 RAG의 출처 인용과 결합

**최종 목표**: RAG의 근거 제시(Groundedness) + Opus의 완전성(Completeness)·실용성(Actionability)을 결합한 하이브리드 시스템.

**2. YouTube 콘텐츠가 커버하는 질문 유형**

realestate_v3의 11,834개 문서는 아래 채널에서 수집된 전문가 콘텐츠:

**3. 할루시네이션 통제 유지의 이유**

- 더 많은 관련 문서가 제공되어 LLM이 문서 기반 답변 생성 가능
- 검색 성공률 상승 → 재검색(rewrite) 빈도 감소 → 불필요한 LLM 호출 감소
- 기존 check_hallucination 노드가 확장된 문서에 대해서도 동일하게 작동

### 10-5. 잔여 35% fail 분석

128/197 pass(65%)에도 불구하고 69개 질문이 여전히 fail:

| fail 유형 | 예상 비율 | 원인 |
|----------|---------|------|
| 시세 예측/타이밍 | ~30% | "지금 사야 할까요" 류 — 문서로 답변 불가 |

| 정부 정책 최신 | ~20% | 2026년 최신 정책 변경 — 문서 시점 이후 |
| 절차적 세부 사항 | ~15% | 건축허가 절차 등 — 실무 디테일 부족 |
| 기타 | ~10% | 생활 정보 (인테리어비, 관리비 등) |
