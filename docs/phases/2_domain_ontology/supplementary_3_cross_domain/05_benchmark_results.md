## 2. Set B — 핵심 목표 달성

2개 기존 실패 질의가 **모두 해결**되었다. 이것이 P1-a의 핵심 성공 기준이다.

| 질의 | Baseline Top-3 | Full Pipeline Top-3 | LLM 분석 유형 |
|------|---------------|-------------------|-------------|
| "부동산 사면 나라에 돈 내야 되나" | 공과금, 속지주의, 국제이중과세 | **취득세 납세의무자, 조세주체, 취득세** | REWRITE |

## 3. Set A — 회귀 분석

### 3-1. 회귀 3건 상세

| 질의 | Baseline 결과 (P@3) | Full 결과 (P@3) | LLM 분석 유형 | 원인 |
|------|---------------------|-----------------|-------------|------|
| "경매 낙찰되면 세금 내야해?" | **O** (낙찰 Top-2) | **X** | DECOMPOSE | LLM이 분해하면서 원본보다 넓은 범위를 검색. 기대 키워드("경매","낙찰","세금","취득")가 Top-3에서 빠짐 |
| "집 계약하고 내 이름으로 바꾸는 절차" | **O** (등기 Top-3) | **X** | REWRITE | LLM이 변환한 질의가 "소유권이전등기 신청 절차" 등으로 바뀌었으나 기대 키워드 미매칭 |
| "부동산 거래할 때 중간에 사기 안 당하려면" | **O** (사기 Top-3) | **X** | REWRITE | LLM 변환 결과가 기대 키워드 "계약,등기,확인,사기,검증"과 미매칭 |

### 3-2. 신규 해결 1건

| 질의 | Baseline 결과 | Full 결과 | LLM 분석 유형 |
|------|-------------|----------|-------------|
| "은행에서 집값의 몇 프로까지 빌려주는지" | **X** (시장이자율) | **O** (담보 Top-3) | REWRITE |

### 3-3. 회귀 원인 분석

**핵심 문제**: LLM이 이미 Baseline에서 성공하던 질의를 **불필요하게 변환**하여 결과가 달라진다.

> **근본 원인 — "과잉 변환"(Over-rewriting):**
>
> 현재 설계에서 LLM은 Tier 2로 넘어온 모든 질의를 변환한다. 그러나 "경매 낙찰되면 세금 내야해?"는 이미 "경매", "낙찰", "세금"이라는 키워드가 BGE-M3 Sparse 벡터에 의해 잘 매칭되고 있었다. LLM이 이를 "부동산 경매 낙찰 절차와 낙찰자 의무" 등으로 분해하면서, 원래 잘 잡히던 키워드가 희석되었다.
>
> 이는 `Scaling RAG Fusion (2026.03)` 논문의 경고 — "RAG Fusion 개선이 프로덕션 re-ranking + context budget 하에서 대부분 소멸한다" — 와 일치하는 현상이다.

## 7. 핵심 발견 요약

### 7-1. 긍정적 발견

1. **LLM Query Rewriting의 효과 입증**: 극단적 구어체("나라에 돈 내야 되나" → "취득세 납부 의무")를 전문용어로 정확히 변환하여 검색 품질을 획기적으로 개선
2. **Avg Top-1 개선**: Set A에서 0.7907 → 0.8445로 Top-1 스코어가 향상. 순위는 변동되었지만 의미적 매칭 품질은 개선
3. **Set D Avg Top-1 대폭 개선**: 0.6356 → 0.8467 (+33%). 구어체 질의의 전문용어 변환이 임베딩 유사도를 크게 높임

### 7-2. 문제점

1. **과잉 변환(Over-rewriting)**: Baseline에서 이미 성공하던 질의를 LLM이 불필요하게 변환하여 회귀 발생
2. **사전 필터 비효율**: LLM 호출율 88%로 목표(60%) 대비 높음. 사전 필터가 컨테이너 경로 문제로 부분적으로만 작동
3. **DECOMPOSE 회귀**: Cross-Domain 분해가 오히려 결과를 악화시키는 경우 다수. 서브 질의별 검색 결과의 RRF 합산이 원본 단일 검색보다 나쁜 경우가 많음
4. **레이턴시**: Claude Code CLI 호출(~4~7초)이 서비스 수준에 맞지 않음. API 직접 호출(~400ms)로 전환 필요

## 8. 다음 단계 과제 (보완 방향)

### P1-a-1: 하이브리드 전략 — 원본 + 변환 질의 동시 검색 (최우선)

> **핵심 아이디어**: LLM 변환 결과와 원본 질의를 **모두** 검색하여 합산. 변환이 좋으면 결과가 개선되고, 변환이 나쁘더라도 원본 결과가 보존된다.

```
[현재]
Query → LLM → 변환된 질의만 검색

[개선]
Query → LLM → 변환된 질의 검색
  │                                → RRF 합산 → Top-K
  └────────→ 원본 질의도 검색
```

**기대 효과**: 회귀 0건 (원본 결과가 항상 합산에 포함되므로). Baseline에서 성공하던 질의가 최소한 동일 수준을 유지.

**구현 난이도**: 낮음 — `pipeline.py`에서 원본 질의 검색 결과를 추가하고 `rrf_merge()`에 포함시키면 됨.

### P1-a-2: 사전 필터 정교화

1. **SIMPLE 판정 조건 완화**: 현재 `len ≤ 15 AND 정규 용어 매칭 AND 단일 도메인`인 조건을 조정하여 더 많은 질의를 SIMPLE로 처리. 특히 2개 이상의 정규 용어가 포함된 질의는 이미 전문적이므로 SIMPLE로 판정.
2. **도메인 키워드 확장**: `_detect_domains()`의 키워드를 온톨로지 데이터에서 자동 생성하여 정확도 향상.

**기대 효과**: LLM 호출율 88% → 50~60%로 감소.

### P1-a-3: 프롬프트 보수성 강화

1. **SIMPLE 판정 확대**: 프롬프트에 "이미 전문용어가 포함된 질의는 SIMPLE로 판정하세요" 지침 추가
2. **DECOMPOSE 억제**: "단일 검색으로 충분한 경우 DECOMPOSE하지 마세요. 질의에 2개 이상의 명확히 다른 도메인이 요구될 때만 DECOMPOSE하세요" 지침 강화
3. **변환 보수성**: "변환 시 원본 질의의 핵심 키워드를 반드시 포함하세요" 지침 추가

### P1-a-4: API 직접 호출로 전환

Claude Code CLI (~4~7초/호출) → Anthropic API 직접 호출 (~400ms/호출)로 전환하여 레이턴시를 서비스 수준으로 낮춤. 현재 API 크레딧 충전 후 적용.

### P1-b: Cross-Encoder 재도입

P1-a 보완 완료 후, Query Rewriting으로 개선된 후보 풀에 Cross-Encoder(Setting E)를 2차 리랭커로 적용. `reranker.py`의 `rerank_results()` 함수를 `pipeline.py`에 선택적 통합.
