## 3. Set B 집중 분석 — 46% Baseline이 확인하는 LLM 필요성

### 3-1. Set B의 핵심 문제

Set B(극단 구어체, 50개)의 P@3 46.0%는 **LLM Query Rewriting 없이는 절반 이상의 구어체 질의를 해결할 수 없음**을 확인한다.

| 유형 | 예시 질의 | 기대 결과 | 실패 원인 |
|------|----------|----------|----------|
| 어휘 격차 | "나라에 돈 내야 되나" | 취득세 | "나라에 돈" ↔ "취득세" 의미적 거리 |
| 비유적 표현 | "몇 프로까지 빌려주는지" | LTV/담보인정비율 | "프로" ↔ "비율", "빌려주는" ↔ "대출한도" 격차 |
| 슬랭 | "영끌해도 되나" | 대출한도/리스크 | 인터넷 슬랭이 임베딩 공간에 부재 |
| 생략 표현 | "세금 얼마야" | 취득세/양도세 | 맥락 없는 축약 표현 |

### 3-2. 이전 42개 벤치마크에서의 LLM 효과 (참고)

이전 42개 질의 벤치마크(Phase 2 Legacy, 아래 섹션 7 참조)에서 Set B(2개 실패 질의)가 LLM Query Rewriting으로 100% 해결된 바 있다. 500개 확장에서도 Set B의 나머지 질의들에 대해 동일한 효과가 기대된다.

## 4. Set C 분석 — 정규 용어가 있으면 크로스 도메인도 성공

### 4-1. Set C의 강점

Set C(크로스 도메인, 100개)의 P@3 **81.0%**는 전 세트 중 최고이다. 정규 용어가 포함되어 있으면 두 도메인을 교차하는 질의에서도 Dense+Sparse RRF가 효과적으로 작동한다.

| 질의 유형 | 예시 | P@3 경향 |
|----------|------|---------|
| 정규+정규 교차 | "재건축 아파트 양도세" | 높음 — 양 도메인 키워드 매칭 |
| 정규+구어체 교차 | "경매 집 사면 세금 얼마야" | 보통 — 한쪽 키워드로 부분 매칭 |
| 구어체+구어체 교차 | "집 팔고 남은 돈에서 떼가나" | 낮음 — Set B/D와 동일 문제 |

### 4-2. 시사점

크로스 도메인 질의의 핵심 성공 요인은 **도메인 교차 자체가 아니라 정규 용어의 존재 여부**이다. Set C의 81%는 대부분 정규 용어가 포함된 질의에서 달성된 것이며, 구어체만으로 구성된 크로스 도메인 질의는 Set B/D와 유사한 패턴으로 실패한다.

## 5. 500개 질의 Baseline에서의 핵심 발견

### 5-1. Baseline P@3 하락 분석

| 비교 | P@3 | 분석 |
|------|-----|------|
| 이전 25개 질의 (Set A 기존) | 92% | 정규 질의 위주로 구성 |
| 500개 전체 | 65.8% | 극단 구어체(B+D)와 슬랭(E) 포함으로 하락 |
| 500개 중 정규+크로스 (A+C) | 80.0% | 이전 25개와 유사한 난이도 질의는 여전히 양호 |
| 500개 중 구어체+슬랭 (B+D+E) | 51.6% | **핵심 병목 구간** |

### 5-2. 세트별 핵심 수치

| 발견 | 세트 | P@3 | 의미 |
|------|------|-----|------|
| Set A baseline 하락 | A | 79.3% | 이전 92% 대비 -12.7%p. A26~A150에 난이도 높은 질의 포함 |
| 크로스 도메인 양호 | C | 81.0% | 정규 용어가 있으면 도메인 교차에도 검색 성공 |
| 극단 구어체 병목 | B+D | ~48% | **P1-a Query Rewriting의 핵심 타겟** |
| 혼합형 중간 | E | 56.0% | 슬랭 질의가 성능 저하의 주 원인 |

### 5-3. P1-a Query Rewriting 필요성 재확인

500개 질의에서 구어체/슬랭 질의(B+D+E = 250개)의 Baseline P@3가 **51.6%**로 확인되었다. 이는:
- 전체 500개 질의의 **절반**이 구어체/슬랭으로 구성
- 해당 구간에서 **절반 이상**이 실패
- 세 가지 검색 설정(C, E, F) 모두에서 **공통적으로** 실패

→ **검색 설정 변경으로는 해결 불가, LLM Query Rewriting이 필수**임을 500개 규모에서 재확인.
