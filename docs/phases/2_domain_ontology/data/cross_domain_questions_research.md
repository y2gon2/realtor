실제 사용자 질의에서 도메인이 교차하는 근본 이유를 6가지로 분류:

| 패턴 ID | 패턴명 | 설명 | 예시 |
|---------|-------|------|------|
| **P1** | 거래 세금 연쇄 | 취득/보유/양도 각 단계에 세금이 따라붙음 | "경매 낙찰 → 취득세", "상속 → 상속세 → 등기" |
| **P2** | 대출-규제 연동 | 규제지역/다주택 여부가 대출 조건을 좌우 | "투기과열지구 → LTV 제한", "다주택 → DSR 강화" |
| **P3** | 권리-배당 충돌 | 등기상 권리순위가 경매 배당을 결정 | "근저당 vs 보증금", "확정일자 vs 당해세" |
| **P4** | 자격-혜택 조건 | 자격 조건 충족 시 세금/대출 혜택 발생 | "생애최초 → 취득세 면제", "신혼부부 → 디딤돌 저리" |
| **P5** | 용도-규제 전환 | 토지 용도 변경 시 규제와 세금이 동시 변동 | "농지전용 → 허가+부담금+취득세", "주거용 전환 → 주택수 포함" |
| **P6** | 법률 관계 연쇄 | 이혼/상속 등 법률 이벤트가 세금+등기+대출에 파급 | "이혼 → 재산분할 → 등기 → 취득세 면제?", "상속 → 주택수 → 종부세" |

## 14. RAG 시스템 시사점

### 14-1. DECOMPOSE가 필요한 전형적 구조

```
"[이벤트/조건] + [도메인A 질문] + [도메인B 질문]"

예: "경매 낙찰되면(이벤트) + 세금 얼마야(TAX) + 등기는 어떻게(REGIS)"
→ Sub-Q1: "경매 낙찰 부동산 취득세 세율" (AUC+TAX)
→ Sub-Q2: "경매 낙찰 후 소유권이전등기 절차" (AUC+REGIS)
```

### 14-2. REWRITE가 필요한 전형적 구조

```
"[구어체 이벤트] + [구어체 결과]"

예: "집주인이 세금 안 내서 집 넘어가면(구어체) 보증금 어떻게 돼(구어체)"
→ Rewrite: "임대인 세금 체납으로 인한 공매 시 임차인 보증금 배당 순위"
```

### 14-3. 사전 필터로 SIMPLE 처리 가능한 구조

```
"[전문 용어 1] + [전문 용어 2]"

예: "조합원입주권 양도소득세" → 이미 전문용어라 원본 그대로 검색
```

### 14-4. 온톨로지 확장 필요 영역

이 질의 세트에서 현재 온톨로지에 부족할 수 있는 개념:
- **부담부증여** (증여+대출 교차): TAX 브랜치에 추가 필요
- **이월과세** (증여 후 양도 시): TAX 브랜치에 추가 필요
- **당해세/조세우선권** (세금+배당 교차): TAX+AUC 교차 개념
- **임차권등기명령** (등기+임대 교차): REGIS+RENT 교차 개념
- **사해행위취소** (계약+법적구제): CON 브랜치에 추가 필요
- **겸용주택 안분** (상가+주택 세금 분리): TAX 브랜치에 추가 필요
- **혼인증여공제** (신혼+증여 세금): TAX 브랜치에 추가 필요
