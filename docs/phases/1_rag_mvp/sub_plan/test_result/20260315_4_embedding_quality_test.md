| **Overlap@10 평균** | **0%** | 낮을수록 좋음 | **우수** |

### 비유사 쌍 상세 분석

- 4쌍 모두 cos_sim < 0.5 기준을 충족
- **Overlap@5, @10 모두 0%** — 비유사 주제 간 검색 결과가 완전히 분리됨
- #3 재건축 초과이익환수제 vs 월세 세액공제 (0.4851): 둘 다 "부동산 세금/부담금" 관련이라 상대적으로 가장 높지만 여전히 0.5 미만
- #2 강남구 아파트 시세 vs 농지 전용 허가 절차 (0.3191): 주거 vs 농지로 가장 낮은 유사도

**점수 순서: hyde > atomic_fact > summary**

- **hyde (0.7537):** HyDE 방식으로 생성된 예상 질문이 자연어 질의와 가장 잘 정렬됨. 이는 HyDE가 "사용자가 이런 질문을 할 것이다"라는 형태로 임베딩되어 있기 때문에 자연어 질의와 코사인 유사도가 높음
- **atomic_fact (0.7249):** 개별 사실 단위로 분리된 청크. 구체적 정보가 담겨 있어 관련 질의와의 매칭 점수가 준수함
- **summary (0.6685):** 문서 전체를 요약한 청크. 포괄적인 내용을 담고 있어 특정 질의와의 집중도가 상대적으로 낮음. 다만 LLM 컨텍스트로 사용 시 종합적 맥락 제공에 유리

**empty_count = 0:** 모든 chunk_type에서 모든 질의에 대해 검색 결과가 반환됨. 데이터 커버리지 양호.

**top1 vs top3 점수 차이:**
- hyde: 0.7537 → 0.7371 (차이 0.0166) — top-1과 top-3 간 점수 하락이 가장 작음 → 상위 결과 간 품질이 균일
- atomic_fact: 0.7249 → 0.7119 (차이 0.0130)
- summary: 0.6685 → 0.6546 (차이 0.0139)

| 등급 | 판별력 (cosine) | 의미 |
|---|---|---|
| **우수** | **0.3684** (> 0.15) | **임베딩이 도메인 의미를 잘 구분함. 다음 단계 진행 가능** |

### 6-3. 강점

1. **판별력 우수 (0.3684):** 기준치 0.15의 2.5배. 유사/비유사 질의 간 의미적 거리가 명확히 구분됨
2. **비유사 쌍 완전 분리:** Overlap@5/10 모두 0% — 서로 다른 주제의 질의가 완전히 다른 문서를 반환
3. **cos_sim 기준 충족:** 유사 쌍 평균 0.7606 (> 0.7), 비유사 쌍 평균 0.3922 (< 0.5)
4. **chunk_type 설계 의도 부합:** hyde가 자연어 질의에 가장 높은 점수로, 설계 의도대로 동작
5. **데이터 커버리지 양호:** 모든 chunk_type에서 empty_count = 0

### 6-4. 약점 및 주의사항

1. **Overlap@5/10 미달:** 유사 쌍의 실제 검색 결과 겹침이 기준보다 낮음 (Overlap@5: 20% vs 기준 40%, Overlap@10: 26% vs 기준 30%)
   - **근본 원인 (키워드 문서풀 분석 결과):** 문서 부족이 아님. 키워드 문서풀은 15~20개로 충분. **질의B(구어체/일상 표현)가 키워드 의미 공간을 벗어나 다른 문서 집합을 검색하는 것이 주된 원인.** 키워드↔질의B 유사도 갭이 클수록 Overlap이 낮아지는 강한 상관관계 확인.
   - **원인 유형별 분류:**
     - 질의B 의미 이탈 (4쌍: #3, #4, #6, #7): 키워드 문서 충분하나 구어체가 다른 의미 공간 검색
     - 키워드 문서풀 집중도 부족 (2쌍: #5, #8): 주제 자체가 넓어 키워드 검색 결과가 산만
     - 복합 원인 (1쌍: #2): 넓은 주제 + "전망" 의도 추가
   - **대응:** (1) query expansion으로 구어체→전문용어 매핑, (2) reranker로 의미적 재정렬, (3) top-K 확대

2. **전문 용어 vs 일상 표현 간 임베딩 거리:** 키워드↔질의B 유사도 갭이 0.09~0.31까지 분포. 특히 약어/전문용어 쌍에서 갭이 큼
   - "초과이익환수제" ↔ "부담금" — 갭 0.30, overlap@10: 20%
   - "DSR" ↔ "총부채원리금상환비율" — 갭 0.31, overlap@10: 10%
   - "다주택자 세금 중과" ↔ "집 여러 채 가진 사람 세금 불이익" — 갭 0.23, overlap@10: 20%
   - **대응:** query expansion, synonym mapping, 또는 LLM 기반 질의 정규화(query normalization)로 보완

**결론:** 임베딩 품질 평가 결과 **"우수"** 등급. KURE-v1은 부동산 도메인에서 의미적 구분을 잘 수행하며, **다음 단계(LLM 연동 RAG 파이프라인)로 진행 가능.**

## 7. Overlap 저조 개선 방안 상세

### 7-0. 문제 정의

```
사용자 구어체 질의 → 임베딩 → 전문용어 기반 문서와 의미 공간 불일치 → 다른 문서 검색
```

Overlap 저조의 근본 원인은 두 가지로 분류된다:

| 원인 유형 | 해당 쌍 | 핵심 문제 |
|---|---|---|
| 질의B 의미 이탈 | #3, #4, #6, #7 | 구어체/일상 표현이 전문용어와 다른 의미 공간에 임베딩됨 |
| 키워드 문서풀 집중도 부족 | #5, #8 | 주제 자체가 넓어 키워드 검색 결과가 산만 |

아래 4가지 방안을 효과/실현성 기준으로 분석한다.

---

### 7-1. 질의 전 LLM 전처리 (Query Normalization)

**키워드↔질의B 갭 자체를 줄이는 가장 직접적인 해결책.**

#### 필요한 데이터

| 항목 | 상세 | 비고 |
|---|---|---|
| **부동산 도메인 용어 사전** | 약어↔풀네임, 전문용어↔일상표현 매핑 테이블 | LLM 프롬프트에 참조용으로 제공 |
| **few-shot 예시** | 구어체→전문용어 변환 입출력 예시 5~10쌍 | 프롬프트 내 포함 |
| **LLM** | 경량 모델 (Haiku급) — latency 최소화 | RAG 파이프라인에 이미 LLM 사용 중 |

#### 용어 사전 예시

```
# 약어 ↔ 풀네임
DSR ↔ 총부채원리금상환비율
LTV ↔ 주택담보대출비율 (담보인정비율)
DTI ↔ 총부채상환비율

# 전문용어 ↔ 일상표현
초과이익환수제 ↔ 재건축 부담금
중과세율 ↔ 세금 불이익, 세금 더 내는 것
다주택자 ↔ 집 여러 채 가진 사람
생애최초 주택구입 ↔ 처음 집 사는 것
경매 낙찰 ↔ 법원에서 집 사는 것
청약 ↔ 새 아파트 분양 신청
```

#### 프로세스

```
사용자 질의: "집 여러 채 가진 사람 세금 불이익"
    ↓ LLM (경량 모델, 용어 사전 참조)
정규화 질의: "다주택자 양도소득세 취득세 종합부동산세 중과세율"
    ↓ 임베딩 → 검색
```

#### 예상 효과

- 키워드↔질의B 갭 0.31 → 0.05~0.10 수준으로 축소 예상
- #3, #4, #6, #7 (의미 이탈 유형)에 직접적 효과

#### 비용/리스크

- 매 질의마다 LLM 호출 1회 추가 → latency + 비용 증가
- 정규화 과정에서 원래 의도가 왜곡될 가능성 → few-shot 품질이 중요

---

### 7-2. Multi-Query Retrieval (질의 확장 검색)

질의를 정규화하는 대신, **여러 변형을 동시에 검색**하여 결과를 합치는 방법.
방법 7-1과 통합하여 1회 LLM 호출로 정규화 + 변형을 동시에 생성할 수 있다.

#### 필요한 데이터

| 항목 | 상세 | 비고 |
|---|---|---|
| **LLM** | 변형 질의 2~3개 생성 | 7-1과 동일 호출에서 처리 가능 |
| **알고리즘** | 결과 합산 방식 — RRF (Reciprocal Rank Fusion) 또는 max-score merge | 구현 필요 |
| **추가 데이터** | 불필요 (LLM이 자체적으로 변형 생성) | |

#### 프로세스

```
원본 질의: "집 여러 채 가진 사람 세금 불이익"
    ↓ LLM (1회 호출)
정규화: "다주택자 세금 중과"
변형 1: "다주택자 양도소득세 중과세율"
변형 2: "2주택 3주택 취득세 종부세 중과"
    ↓ 각각 임베딩 → 검색 (top-K씩)
    ↓ 결과 합산 (RRF)
    ↓ 최종 top-K 반환
```

#### 예상 효과

- 7-1보다 recall이 높음 — 원본 구어체가 잡는 문서 + 전문용어가 잡는 문서 모두 확보
- #5, #8 (문서풀 산만 유형)에도 효과 — 다양한 변형이 넓은 문서풀을 커버

#### 비용/리스크

- 벡터 검색 횟수 3~4배 증가 (단, 벡터 검색 자체는 ms 단위로 빠름)
- LLM 호출은 7-1과 합쳐 1회로 처리 가능

---

### 7-3. Reranker (Cross-Encoder 기반 재정렬)

검색 후 단계에서 질의-문서 쌍의 관련성을 재평가. 방법 7-1, 7-2와 **병행 사용** 가능.

#### 필요한 데이터

| 항목 | 상세 | 비고 |
|---|---|---|
| **한국어 Cross-Encoder 모델** | `ko-reranker`, `bge-reranker-v2-m3` 등 | 모델 선정 및 평가 필요 |
| **GPU 추론 인프라** | 기존 embedding 서버에 추가 로드 또는 별도 서버 | 추론 비용 발생 |
| **학습 데이터** | pretrained 모델 사용 시 불필요. 부동산 도메인 파인튜닝 시 질의-문서 관련성 쌍 필요 | 선택사항 |

#### 프로세스

```
질의 → 임베딩 검색 (top-20~50 확보)
    ↓
Cross-Encoder가 (질의, 문서) 쌍마다 관련성 점수 산출
    ↓
재정렬된 top-K 반환
```

#### Overlap 문제 해결 원리

- **Bi-encoder(KURE-v1):** 질의와 문서를 **독립적으로** 임베딩 → "부담금" ↔ "초과이익환수제" 같은 간접적 의미 관계를 놓침
- **Cross-encoder:** 질의와 문서를 **동시에** 입력받아 상호 참조 → "재건축 부담금이 뭔가요?" 질의와 "초과이익환수제" 문서 내용을 함께 보고 관련성 판단 가능

#### 예상 효과

- 7-1/7-2로 확보한 후보군의 순서를 최적화
- 구어체 질의와 전문용어 문서 간 매칭 정확도 향상

#### 비용/리스크

- 검색 범위를 넓혀야(top-20~50) 의미가 있음 → 초기 검색 비용 증가
- Cross-encoder 추론 비용 (top-30 문서 × 추론 1회씩)

---

### 7-4. 문서 측 보강 (인덱싱 단계)

질의를 바꾸는 대신, **문서 쪽에 구어체 표현을 추가**하는 방법.

#### 선택지

| 방식 | 설명 | 필요 데이터 |
|---|---|---|
| **구어체 HyDE 추가** | 기존 hyde 청크 외에, "일반인이 할 법한 질문"을 추가 생성하여 색인 | LLM으로 문서당 2~3개 생성 |
| **synonym 메타데이터** | 문서 내 전문용어에 대한 일상 표현을 메타데이터로 추가 | 용어 사전 + LLM 태깅 |

#### 필요한 데이터

| 항목 | 상세 | 비고 |
|---|---|---|
| **LLM** | 구어체 HyDE 질문 생성 (문서당 2~3개) | 일회성 배치 작업 |
| **스토리지** | 포인트 수 증가 (현재 93,943 → +12,000~18,000 예상) | Qdrant 스토리지 |
| **리인덱싱** | chunker → embedder → upserter 파이프라인 재실행 | 일회성 |

#### 예상 효과

- 일회성 작업으로 영구적 효과 — 질의 시 추가 비용 없음
- 구어체 사용자가 별도 전처리 없이도 관련 문서에 도달 가능

#### 비용/리스크

- 포인트 수 증가에 따른 검색 성능/비용 고려
- 현재 hyde 청크의 질문 스타일 점검 필요 — 이미 구어체가 포함되어 있다면 불필요
- 리인덱싱 작업 시간

### 7-5. 권장 적용 순서 및 파이프라인 설계

단독 적용보다 **조합**이 효과적. 비용 대비 효과 순으로 단계별 적용을 권장한다.

#### 전체 파이프라인

```
[사용자 질의 입력]
    ↓
① LLM 질의 정규화 + 변형 생성 (7-1 + 7-2 통합, LLM 1회 호출)
   - 정규화 질의 1개 + 변형 2개 생성
   - 비용: 경량 LLM 1회
    ↓
② 다중 질의 벡터 검색 (원본 + 정규화 + 변형 각 top-K)
   - 비용: 벡터 검색 3~4회 (ms 단위)
    ↓
③ 결과 합산 (RRF)
   - 비용: 무시 가능
    ↓
④ (선택) Reranker 재정렬
   - 비용: Cross-encoder 추론 (top-20~30 문서)
    ↓
[최종 top-K → LLM 답변 생성]
```

#### 단계별 적용 우선순위

| 단계 | 방안 | 이유 | 추가 필요사항 | 예상 효과 |
|---|---|---|---|---|
| **1단계** | LLM 질의 정규화 + Multi-Query (7-1 + 7-2) | 추가 데이터/모델 불필요, LLM만 있으면 즉시 적용 가능 | 프롬프트 설계, 용어 사전 작성, RRF 합산 로직 구현 | 키워드↔B 갭 0.31 → 0.05~0.10 축소. Overlap 대폭 개선 예상 |
| **2단계** | Reranker 도입 (7-3) | 1단계로 확보한 후보군의 순서 최적화 | 한국어 Cross-Encoder 모델 선정, GPU 추론 인프라 | 구어체/전문용어 간 순위 편차 보완 |
| **3단계** | 구어체 HyDE 보강 (7-4) | 질의 전처리 없이도 구어체 매칭 개선 (영구적) | 리인덱싱 파이프라인 실행, 기존 hyde 스타일 분석 | 질의 시 추가 비용 없는 근본적 개선 |

> **1단계만으로도 Overlap 문제의 대부분을 해결할 수 있을 것으로 예상.** 예: "집 여러 채 가진 사람 세금 불이익" → LLM이 "다주택자 세금 중과"로 정규화하면, 키워드↔질의B 갭이 사실상 0에 근접.
