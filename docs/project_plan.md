## 1. 프로젝트 개요 및 비전

### 1.1 서비스 컨셉

**"주소 하나를 입력하면, 그 부동산에 대한 사람이 쓴 것처럼 완결된 맞춤형 분석 보고서를 제공하는 AI 어드바이저"**

- 사용자는 주소(또는 단지명)를 입력한다.
- LLM이 보고서 작성에 필요한 추가 정보(목적, 재정 상황, 주택 수 등)를 **대화형으로 인터뷰**한다.
  - 선택형: 매매/전세/월세, 실거주/투자/임대, 무주택/1주택/다주택 등
  - 주관식: 예상 대출액, 보유 기간, 월 상환 가능액 등
- 수집된 정보를 바탕으로 **Deep Research 스타일의 구조화된 맞춤 보고서**를 생성한다.
- 경매·공매 물건도 커버 가능하도록 확장.

### 1.2 핵심 가치 제안

| 기존 서비스 | 본 서비스 |
|---|---|
| 매물 탐색/추천 | 특정 물건 심층 분석 |
| 시세 조회 | 정책·거래·상권·리스크 통합 보고서 |
| 단발 Q&A 답변 | 인터뷰 → 사용자 조건 반영 완결 리포트 |
| 출처 없는 요약 | RAG 근거 문장 인용 + 룰 기반 정책 판정 |

### 1.3 주요 보고서 구성 (목표 출력물)

```
[부동산 딥 리포트 — {주소} — {날짜}]

① 요약 결론 (한 줄 + 근거 링크)
② 정책 영향 카드
   - 대출(LTV/DSR 적용 결과)
   - 세금(취득세/보유세/양도세 시나리오)
   - 임대차(계약갱신/전월세전환율)
   - 정비사업·재건축 여부
③ 거래 동향 (최근 12개월 실거래, 가격 분포)
④ 주변 시세 (반경 1km 유사 단지 비교)
⑤ 상권 분석 (업종 밀도, 접근성, 수요 프록시)
⑥ 체크리스트 (현장 확인 / 권리 리스크 / 자금 계획)
```

## 3. 현재 데이터의 RAG 활용 가능성 평가

### 3.1 결론 요약

**"활용 가능하다. 단, 단독으로는 부족하며 구조화 데이터와 반드시 결합해야 한다."**

### 3.2 현재 데이터의 강점 (RAG에 적합한 이유)

1. **"왜(Why)" 레이어를 담당** — 정책의 취지, 시장 해석, 투자자 경험, 사례 기반 판단 근거
2. **원자 사실(Atomic Facts) 구조** — 청킹(chunking) 없이 바로 검색 단위로 활용 가능
3. **v2 HyDE 질문 포함** — 예상 쿼리와 문서 간 임베딩 정렬이 이미 내재됨
4. **시계열 다양성** — 다양한 시장 사이클의 해석을 제공
5. **다채널 관점** — 세금 전문, 경매 전문, 지역 전문 등 도메인 특화

### 3.3 현재 데이터의 한계 (RAG만으로 불가한 것들)

| 질문 유형 | 현재 데이터로 답 가능? | 이유 |
|---|---|---|
| "이 아파트 지금 얼마야?" | ❌ | 실거래 DB 없음 |
| "내 대출 한도는?" | ❌ (부분) | 사용자 조건 변수 계산 불가 |
| "주변 학교까지 거리는?" | ❌ | 공간 데이터 없음 |
| "최근 3개월 거래량은?" | ❌ | 정량 데이터 없음 |
| "DSR 2단계 적용받나?" | ⚠️ | 정책 설명은 있으나 룰 계산 불가 |
| "재건축 가능성은?" | ⚠️ | 일반론은 있으나 해당 단지 구체 정보 없음 |
| "이 규정은 언제 바뀌었나?" | ✅ (일부) | 날짜 메타데이터로 버전 관리 가능 |
| "투자자들은 이 지역을 어떻게 보나?" | ✅ | 핵심 강점 |

### 3.4 RAG에서의 역할 정의

```
현재 데이터(YouTube 노트) = "해석·판단·맥락 레이어"
구조화 DB(공공 API) = "숫자·사실·현황 레이어"
정책 룰 엔진 = "적용 가능 여부 판정 레이어"

→ 세 레이어가 결합될 때 비로소 "결정에 쓸 수 있는" 보고서가 나온다
```

## 4. RAG 데이터베이스 구축 전략

### 4.1 문서 파이프라인 (현재 → 목표)

```
[원본 스크립트] (notes_rag_todo, 32,631개)
    ↓ LLM 변환 (현재 진행 중)
[v1 구조화 노트] (notes_rag_done, 11,296개)
    ↓ LLM 정제
[v2 정제 노트] (rag_refined, 558개)  ← 색인 목표 대상
    ↓ 메타데이터 보강 + 청킹
[RAG 색인 레코드]
```

**목표**: 최소 20,000개 v2 정제 문서 완성 후 색인 시작

### 4.2 메타데이터 스키마 (색인 기준)

v2 포맷에 아래 필드 추가/보완 필요:

```yaml
# 기존 (유지)
doc_type: youtube_study_note
source.upload_date: "2024-11-03"
source.channel: "weolbu_official"

# 추가 필요
topic_tags:          # 주제 분류 (멀티라벨)
  - "대출/LTV/DSR"
  - "정책/규제지역"
region_tags:         # 언급 지역
  - "전국"
  - "서울/강남구"
policy_period:       # 정책 적용 기간 (언급된 경우)
  start: "2024-09-01"
  end: null
reliability_score:   # 출처 신뢰도 등급
  # 1=정부/공식, 2=전문가채널, 3=인플루언서, 4=일반
  level: 2
asset_type:          # 자산 유형
  - "아파트"
  - "경매"
```

### 4.3 청킹(Chunking) 전략

현재 v1/v2 포맷은 이미 **"원자 사실(Atomic Facts)"** 단위로 분리되어 있으므로,
두 가지 청크 레벨을 사용한다:

| 청크 레벨 | 크기 | 용도 |
|---|---|---|
| **문서 청크** | 전체 문서 요약 (~500토큰) | 전체 맥락 파악 |
| **원자 사실 청크** | 단일 Fact (~50~100토큰) | 정밀 사실 검색 |
| **HyDE 청크** | 예상 질문 + 답변 임베딩 | 쿼리-문서 정렬 |

### 4.4 임베딩 및 검색 전략

```
1단계: 하이브리드 검색
   - BM25 키워드 검색 (지명·정책명 등 고유명사 정확도)
   - 벡터 임베딩 검색 (의미 유사도)
   - 결합 점수 = α×BM25 + (1-α)×Vector (α≈0.3~0.5)

2단계: 메타데이터 필터링 (검색 전 pre-filter)
   - 날짜 범위 (최신 정책 질문 → 최근 2년 가중)
   - 지역 태그 (주소 → 행정구역 매핑 후 필터)
   - 주제 태그 (질문 의도 분류 후 필터)

3단계: 리랭킹 (검색 후 re-rank)
   - Cross-encoder 재순위 (정확도 향상)
   - Recency bias: 최신 문서 가중치 부여

4단계: 컨텍스트 조립 (Context Assembly)
   - 상위 K개 청크 선택 (K≈10~15)
   - 출처·날짜 메타데이터 함께 전달
```

### 4.5 운영 파이프라인 (증분 인덱싱)

```
매일/주기적:
  1. 신규 YouTube 스크립트 수집 → notes_rag_todo
  2. LLM 변환 배치 → notes_rag_done (v1)
  3. LLM 정제 배치 → rag_refined (v2)
  4. 중복 제거 (URL 기반 dedup)
  5. 임베딩 생성 → 벡터 DB에 증분 삽입
  6. 버전 As-of 태그 갱신
```

## 5. 추가 필요 데이터 및 공공 API

### 5.1 필수 공공 API (우선순위 순)

#### A. 실거래가 (핵심 — 없으면 보고서의 핵심이 빠짐)

| API | 제공처 | 데이터 |
|---|---|---|
| 아파트 매매 실거래가 조회 | 국토교통부 / data.go.kr | 단지/동/면적/층/계약일/금액 |
| 아파트 전월세 실거래가 | 국토교통부 / data.go.kr | 전세보증금, 월세 |
| 토지 매매 실거래가 | 국토교통부 / data.go.kr | 농지·임야·토지 거래 |

**활용**: 최근 12개월 거래량, 평균가, 분위수, 신고가 비율 계산

#### B. 공간 정보 (주소 → 공간 분석 핵심)

| API | 제공처 | 데이터 |
|---|---|---|
| 주소 → 좌표 변환 | 네이버 지도 API (Geocoding) | 위경도, 행정구역코드 |
| 주변 시설 검색 | 네이버 지도 API (Place Search) | 학교/병원/마트/지하철역 |
| 경로/거리 계산 | 네이버 지도 API (Direction) | 도보·차량 이동시간 |
| 연속지적도/필지 경계 | VWorld WFS | 필지 ID(PNU), 경계 폴리곤 |
| 토지이용계획 | 국토교통부 WMS/WFS | 용도지역·지구·구역 |

#### C. 건물·단지 정보

| API | 제공처 | 데이터 |
|---|---|---|

| 토지임야정보(속성) | 국토교통부 / data.go.kr | 지목, 면적, 공시지가 |

#### D. 금융/대출 계산

| 항목 | 방법 |
|---|---|

| 기준금리 | 한국은행 ECOS API |
| 공동주택 공시가격 | 국토교통부 공시가격 API |

#### E. 세금 계산

### 5.2 고려 가능한 민간/유료 데이터

| 데이터 | 필요성 | 비고 |

## 9. 경쟁사 분석 및 차별화 전략

### 9.1 주요 경쟁사 비교

| 서비스 | 컨셉 | 강점 | 우리와의 차이 |
|---|---|---|---|
| **부동산플래닛 '플래닛AI'** | 대화형 시세/실거래 분석 | 실거래 DB 강함 | 우리: 정책 룰엔진 + RAG 근거 인용 강화 |
| **부톡 '부토기'** | LLM+RAG 대화형 중개 플랫폼 | RAG + 실무 리스크 Q&A | 우리: 중개 아님, 딥 리포트에 집중 |
| **직방 'AI중개사'** | 대화형 매물 추천 | 매물 DB, 거주자 리뷰 | 우리: 탐색 아님, 1개 물건 심층 분석 |
| **네이버페이 'AI 집찾기'** | 자연어 매물 검색 | 플랫폼 트래픽, 지도 연동 | 우리: 보고서 생성 특화 |
| **호갱노노** | 실거래 시각화, 학군 분석 | 데이터 가시화 UX | 우리: 텍스트 보고서 + 해석 레이어 |

### 9.2 핵심 차별화 포인트 (3가지)

**차별화 1: "인터뷰 결과가 보고서 내용을 실제로 바꾼다"**
- 무주택자 vs 1주택자 → 취득세율, 대출 한도, 양도세 시뮬레이션이 달라짐
- 실거주 vs 투자 → 강조 섹션이 달라짐

**차별화 2: "정책 = 룰 판정 + RAG 근거 인용의 이중 구조"**
- 단순 "DSR 40%입니다"가 아니라
- "당신 소득 기준 한도 X만원 (룰 계산) — 출처: 금융위원회 2024.09 고시 + 전문가 채널 해설 (RAG 인용)"

**차별화 3: "주소 1건 완결형 딥 리포트"**
- 경쟁사는 탐색/추천 중심
- 우리는 "이 물건 사도 되는가"에 대한 최종 결론 + 체크리스트

### 9.3 틈새 시장 전략

1. **경매 특화 리포트**: 권리분석 + 명도 리스크 + 낙찰가율 분석 (공매·법원경매)
2. **농지·임야 특화**: 기존 서비스들이 커버 안 하는 영역
3. **B2B 수요**: 중개법인, 법무법인, 자산운용사 대상 보고서 API 제공

| **공인중개사법** | AI가 중개 행위로 해석될 수 있음 | "투자 정보 제공"으로 포지셔닝, 면책 고지 필수 |
| **저작권** | YouTube 스크립트 기반 데이터 | 요약/변환 데이터 활용, 원본 재현 금지, 출처 표기 |
| **금융투자 규제** | 투자 권유로 해석 위험 | "참고 정보" 면책, 투자 결정은 사용자 책임 명시 |
| **개인정보** | 실거래 데이터 일부 개인정보 포함 가능 | 공공 API 범위 내 사용, 개인 식별 데이터 저장 금지 |
| **세금 계산** | 세무사법 위반 소지 | "시뮬레이션 참고용"으로 한정, 세무사 상담 권유 |

### 10.2 기술적 리스크

| 리스크 | 내용 | 대응 방안 |
|---|---|---|
| **LLM 환각(Hallucination)** | 숫자·정책 오류 생성 | 룰 엔진으로 정량 계산, RAG로 근거 제한 |
| **최신성 유지** | 정책이 수시로 바뀜 | 증분 인덱싱 + 정책 DB 주 1회 갱신 |
| **주소 매핑 실패** | 다양한 주소 표기 | 정규화 전처리 + 수동 매핑 케이스 관리 |
| **API 의존성** | 네이버/국토부 API 변경·장애 | 캐시 레이어 + 폴백 처리 |
| **검색 품질 저하** | 데이터 증가 시 노이즈 증가 | 품질 필터링 + 평가셋 회귀 테스트 |

### 10.3 사업적 리스크

| 리스크 | 내용 | 대응 방안 |
|---|---|---|
| **대형 플랫폼 경쟁** | 네이버/카카오의 유사 서비스 출시 | 딥 리포트 / B2B / 경매 특화로 차별화 |
| **데이터 품질** | YouTube 출처의 신뢰도 한계 | 신뢰도 스코어 표시, 공식 출처 우선 |
| **수익 모델** | 무료로 시작하면 API 비용 부담 | 프리미엄 리포트 유료화, B2B API 계약 |

### 10.4 데이터 품질 관리 필수 사항

1. **중복 제거**: 동일 영상 스크립트가 채널별로 중복 존재할 수 있음
2. **날짜 신뢰도**: 업로드일과 내용 기준일이 다를 수 있음 (예: "지난달 정책" 언급)
3. **채널 신뢰도 스코어**: 세금 전문 채널 > 일반 인플루언서 채널
4. **정보 만료 처리**: 1년+ 된 정책 정보에 "현재 적용 여부 확인 필요" 태그

## 11. 객관적 효용성 및 실현 가능성 평가

### 11.1 현재 데이터의 RAG 활용 가능성

| 평가 항목 | 점수 | 근거 |
|---|---|---|
| 데이터 양 | ★★★★☆ | 11,296개 완료, 32,631개 대기 — 충분한 볼륨 |
| 데이터 구조 | ★★★★★ | Atomic Facts + HyDE 구조 — RAG 최적화 설계 |
| 시계열 커버리지 | ★★★★☆ | 2015~2026 다양한 사이클 커버 |
| 주제 다양성 | ★★★★☆ | 세금/경매/임대차/거시/지역 등 폭넓음 |
| 정량 데이터 | ★☆☆☆☆ | 실거래/가격 정량 데이터 없음 (공공 API로 보완 필수) |
| **종합** | **★★★★☆** | RAG 지식 레이어로 즉시 활용 가능 |

### 11.2 프로젝트 실현 가능성

**높은 가능성 측면:**
- 공공 API 인프라 완비 (국토부, VWorld, 캠코 등 무료 API 풍부)
- 한국어 LLM 성능 충분 (Claude, GPT-4o 등)
- 경쟁사 대비 명확한 포지셔닝 가능 (딥 리포트 특화)
- 데이터 파이프라인 이미 구축 중

**어려운 측면:**
- 정책 룰 엔진 초기 구축 비용 높음 (전문 지식 + 수작업)
- 주소→단지 정확 매핑의 엣지 케이스 많음
- 실거래 데이터와 문서 데이터의 결합 로직 복잡도 높음
- 지속적 정책 업데이트 운영 비용

### 11.3 고객 요구 충족 예상도

| 사용자 니즈 | Phase 1에서 충족 가능? | Phase 2 이후? |

| "지금 살 만한가?" | ⚠️ 일반론 수준 | ✅ 사용자 조건 반영 |
| "대출 얼마나 받아?" | ❌ | ✅ 룰 엔진 완성 후 |
| "세금 얼마나 나와?" | ❌ | ✅ 시뮬레이션 가능 |
| "경매 이 물건 어때?" | ⚠️ 일반 경매 지식 | ✅ 공매 API 연동 후 |
| "주변 학교/교통은?" | ❌ | ✅ 공간 분석 모듈 후 |

### 11.4 서비스 효용성 총평

**긍정적 전망:**
- 국내에 "주소 1건 완결형 딥 리포트"를 제공하는 서비스는 현재 없거나 초기 단계
- 부동산 의사결정은 고관여 영역 → 정확하고 종합적인 정보에 높은 지불 의향
- YouTube 기반 코퍼스는 "실제 투자자·전문가의 살아있는 해석"을 담고 있어 다른 서비스가 쉽게 복제 불가

**주의 필요:**
- Phase 1(문서 RAG만)으로 상용 서비스 출시는 리스크 높음 — 최소 Phase 2 완성 후 공개 권장
- 법적 면책 문구 설계를 초기부터 법률 검토와 병행
- "보고서의 정확성"에 대한 사용자 기대 관리 필요 (참고용 명시)
