    3. 현재 데이터의 RAG 활용 가능성 평가¶

    3.1 결론 요약¶

    "활용 가능하다. 단, 단독으로는 부족하며 구조화 데이터와 반드시 결합해야 한다."

    3.2 현재 데이터의 강점 (RAG에 적합한 이유)¶

    1. "왜(Why)" 레이어를 담당 — 정책의 취지, 시장 해석, 투자자 경험, 사례 기반 판단 근거
    2. 원자 사실(Atomic Facts) 구조 — 청킹(chunking) 없이 바로 검색 단위로 활용 가능
    3. v2 HyDE 질문 포함 — 예상 쿼리와 문서 간 임베딩 정렬이 이미 내재됨
    4. 시계열 다양성 — 다양한 시장 사이클의 해석을 제공
    5. 다채널 관점 — 세금 전문, 경매 전문, 지역 전문 등 도메인 특화

    3.3 현재 데이터의 한계 (RAG만으로 불가한 것들)¶

    질문 유형 현재 데이터로 답 가능? 이유
    "이 아파트 지금 얼마야?" ❌ 실거래 DB 없음
    "내 대출 한도는?" ❌ (부분) 사용자 조건 변수 계산 불가
    "주변 학교까지 거리는?" ❌ 공간 데이터 없음
    "최근 3개월 거래량은?" ❌ 정량 데이터 없음
    "DSR 2단계 적용받나?" ⚠️ 정책 설명은 있으나 룰 계산 불가
    "재건축 가능성은?" ⚠️ 일반론은 있으나 해당 단지 구체 정보 없음
    "이 규정은 언제 바뀌었나?" ✅ (일부) 날짜 메타데이터로 버전 관리 가능
    "투자자들은 이 지역을 어떻게 보나?" ✅ 핵심 강점

    3.4 RAG에서의 역할 정의¶

    현재 데이터(YouTube 노트) = "해석·판단·맥락 레이어"
구조화 DB(공공 API) = "숫자·사실·현황 레이어"
정책 룰 엔진 = "적용 가능 여부 판정 레이어"

→ 세 레이어가 결합될 때 비로소 "결정에 쓸 수 있는" 보고서가 나온다

    4.3 청킹(Chunking) 전략¶

    현재 v1/v2 포맷은 이미 "원자 사실(Atomic Facts)" 단위로 분리되어 있으므로, 두 가지 청크 레벨을 사용한다:

    청크 레벨 크기 용도
    문서 청크 전체 문서 요약 (~500토큰) 전체 맥락 파악
    원자 사실 청크 단일 Fact (~50~100토큰) 정밀 사실 검색
    HyDE 청크 예상 질문 + 답변 임베딩 쿼리-문서 정렬

    4.4 임베딩 및 검색 전략¶

    1단계: 하이브리드 검색
   - BM25 키워드 검색 (지명·정책명 등 고유명사 정확도)
   - 벡터 임베딩 검색 (의미 유사도)
   - 결합 점수 = α×BM25 + (1-α)×Vector (α≈0.3~0.5)

2단계: 메타데이터 필터링 (검색 전 pre-filter)
   - 날짜 범위 (최신 정책 질문 → 최근 2년 가중)
   - 지역 태그 (주소 → 행정구역 매핑 후 필터)
   - 주제 태그 (질문 의도 분류 후 필터)

3단계: 리랭킹 (검색 후 re-rank)
   - Cross-encoder 재순위 (정확도 향상)
   - Recency bias: 최신 문서 가중치 부여

4단계: 컨텍스트 조립 (Context Assembly)
   - 상위 K개 청크 선택 (K≈10~15)
   - 출처·날짜 메타데이터 함께 전달

    4.5 운영 파이프라인 (증분 인덱싱)¶

    매일/주기적:
  1. 신규 YouTube 스크립트 수집 → notes_rag_todo
  2. LLM 변환 배치 → notes_rag_done (v1)
  3. LLM 정제 배치 → rag_refined (v2)

    1-7. 평가셋 구축 고정 100개 질문셋으로 RAG 품질 회귀 테스트 정기적 품질 측정 기준

    Phase 2: 구조화 데이터 통합 (5~8개월)¶

    목표: 실거래 DB + 공공 API 연동으로 정량적 보고서 가능

    작업 세부 내용 중요 고려사항
    2-1. 실거래 DB 구축 국토교통부 실거래가 API 수집 + PostgreSQL 저장 최소 2년치 초기 적재, 이후 월별 증분
    2-2. 주소 정규화 파이프라인 사용자 입력 → 법정동코드 + PNU 매핑 주소 오입력, 다양한 표기 처리
    2-3. 공간 분석 모듈 네이버 지도 API + VWorld로 반경 조회 API 비용 관리 (캐싱 전략 필수)
    2-4. 건축물대장 연동 단지 기본정보(건축년도/세대수/주차) 아파트 ID 매핑 정확도 확인
    2-5. 거래 분석 엔진 최근 12개월 거래량/가격분포/신고가비율 계산 거래량 적은 지역 처리 방법
    2-6. 주변 시세 비교 반경 1km 유사 단지 TOP N 비교 (면적·연식 보정) 유사도 기준 명확화 필요
    2-7. 보고서 템플릿 구현 구조화 섹션 + RAG 근거 결합 숫자 섹션 + 해석 섹션 분리

    성공 기준: - 실거래 데이터 정확도 95%+ (API 기준) - 주소→단지 매핑 성공률 90%+ - 거래 동향 섹션 자동 생성

    Phase 3: 정책 룰 엔진 (8~12개월)¶

    목표: 사용자 조건 기반 정책 적용 여부를 룰 기반으로 정확 계산

    작업 세부 내용 중요 고려사항
    3-1. 정책 타임라인 DB 설계 적용기간/적용지역/적용대상/영향범주 스키마 최초 수작업 입력 비용 높음
    3-2. 핵심 정책 입력 LTV/DSR 한도, 취득세율, 양도세율, 규제지역 이력 법령 해석 오류 리스크 주의
    3-3. 룰 평가기 구현 사용자 조건 + 지역 + 날짜 → 정책 적용 결과 경계조건(가격/면적/보유기간) 예외 처리
    3-4. 세금 계산기 통합 취득세/양도세/보유세 시나리오 계산 "법적 조언 아님" 면책 고지 필수
    3-5. RAG + 룰 결합 룰 판정 결과 → 관련 RAG 문서로 근거 보완 두 레이어 결합 프롬프트 설계
    3-6. As-of 버전 지원 "2022년 당시 규정으로 보면?" 요청 처리 정책 이력 테이블 필수

    성공 기준: - 주요 정책(LTV/DSR/취득세) 계산 정확도 95%+ - 정책 변경 시 1주 이내 업데이트 가능

    Phase 4: 서비스화 및 고도화 (12개월+)¶

    9.1 주요 경쟁사 비교¶

    서비스 컨셉 강점 우리와의 차이

    9.2 핵심 차별화 포인트 (3가지)¶

    차별화 1: "인터뷰 결과가 보고서 내용을 실제로 바꾼다" - 무주택자 vs 1주택자 → 취득세율, 대출 한도, 양도세 시뮬레이션이 달라짐 - 실거주 vs 투자 → 강조 섹션이 달라짐

    차별화 2: "정책 = 룰 판정 + RAG 근거 인용의 이중 구조" - 단순 "DSR 40%입니다"가 아니라 - "당신 소득 기준 한도 X만원 (룰 계산) — 출처: 금융위원회 2024.09 고시 + 전문가 채널 해설 (RAG 인용)"

    차별화 3: "주소 1건 완결형 딥 리포트" - 경쟁사는 탐색/추천 중심 - 우리는 "이 물건 사도 되는가"에 대한 최종 결론 + 체크리스트

    9.3 틈새 시장 전략¶

    1. 경매 특화 리포트: 권리분석 + 명도 리스크 + 낙찰가율 분석 (공매·법원경매)
    2. 농지·임야 특화: 기존 서비스들이 커버 안 하는 영역
    3. B2B 수요: 중개법인, 법무법인, 자산운용사 대상 보고서 API 제공

    10. 리스크 및 주요 고려사항¶

    10.1 법적 리스크¶

    리스크 내용 대응 방안
    공인중개사법 AI가 중개 행위로 해석될 수 있음 "투자 정보 제공"으로 포지셔닝, 면책 고지 필수
    저작권 YouTube 스크립트 기반 데이터 요약/변환 데이터 활용, 원본 재현 금지, 출처 표기
    금융투자 규제 투자 권유로 해석 위험 "참고 정보" 면책, 투자 결정은 사용자 책임 명시
    개인정보 실거래 데이터 일부 개인정보 포함 가능 공공 API 범위 내 사용, 개인 식별 데이터 저장 금지
    세금 계산 세무사법 위반 소지 "시뮬레이션 참고용"으로 한정, 세무사 상담 권유

    10.2 기술적 리스크¶

    리스크 내용 대응 방안
    LLM 환각(Hallucination) 숫자·정책 오류 생성 룰 엔진으로 정량 계산, RAG로 근거 제한
    최신성 유지 정책이 수시로 바뀜 증분 인덱싱 + 정책 DB 주 1회 갱신
    주소 매핑 실패 다양한 주소 표기 정규화 전처리 + 수동 매핑 케이스 관리
    API 의존성 네이버/국토부 API 변경·장애 캐시 레이어 + 폴백 처리
    검색 품질 저하 데이터 증가 시 노이즈 증가 품질 필터링 + 평가셋 회귀 테스트

    10.3 사업적 리스크¶

    리스크 내용 대응 방안
    대형 플랫폼 경쟁 네이버/카카오의 유사 서비스 출시 딥 리포트 / B2B / 경매 특화로 차별화
    데이터 품질 YouTube 출처의 신뢰도 한계 신뢰도 스코어 표시, 공식 출처 우선
    수익 모델 무료로 시작하면 API 비용 부담 프리미엄 리포트 유료화, B2B API 계약

    10.4 데이터 품질 관리 필수 사항¶

    1. 중복 제거: 동일 영상 스크립트가 채널별로 중복 존재할 수 있음
    2. 날짜 신뢰도: 업로드일과 내용 기준일이 다를 수 있음 (예: "지난달 정책" 언급)
    3. 채널 신뢰도 스코어: 세금 전문 채널 > 일반 인플루언서 채널
    4. 정보 만료 처리: 1년+ 된 정책 정보에 "현재 적용 여부 확인 필요" 태그

    11. 객관적 효용성 및 실현 가능성 평가¶

    11.1 현재 데이터의 RAG 활용 가능성¶

    평가 항목 점수 근거

    데이터 구조 ★★★★★ Atomic Facts + HyDE 구조 — RAG 최적화 설계

    종합 ★★★★☆ RAG 지식 레이어로 즉시 활용 가능

    11.2 프로젝트 실현 가능성¶

    높은 가능성 측면: - 공공 API 인프라 완비 (국토부, VWorld, 캠코 등 무료 API 풍부) - 한국어 LLM 성능 충분 (Claude, GPT-4o 등) - 경쟁사 대비 명확한 포지셔닝 가능 (딥 리포트 특화) - 데이터 파이프라인 이미 구축 중

    어려운 측면: - 정책 룰 엔진 초기 구축 비용 높음 (전문 지식 + 수작업) - 주소→단지 정확 매핑의 엣지 케이스 많음 - 실거래 데이터와 문서 데이터의 결합 로직 복잡도 높음 - 지속적 정책 업데이트 운영 비용

    11.3 고객 요구 충족 예상도¶

    사용자 니즈 Phase 1에서 충족 가능? Phase 2 이후?

    "지금 살 만한가?" ⚠️ 일반론 수준 ✅ 사용자 조건 반영

    "세금 얼마나 나와?" ❌ ✅ 시뮬레이션 가능
    "경매 이 물건 어때?" ⚠️ 일반 경매 지식 ✅ 공매 API 연동 후
    "주변 학교/교통은?" ❌ ✅ 공간 분석 모듈 후

    11.4 서비스 효용성 총평¶

    긍정적 전망: - 국내에 "주소 1건 완결형 딥 리포트"를 제공하는 서비스는 현재 없거나 초기 단계 - 부동산 의사결정은 고관여 영역 → 정확하고 종합적인 정보에 높은 지불 의향 - YouTube 기반 코퍼스는 "실제 투자자·전문가의 살아있는 해석"을 담고 있어 다른 서비스가 쉽게 복제 불가

    주의 필요: - Phase 1(문서 RAG만)으로 상용 서비스 출시는 리스크 높음 — 최소 Phase 2 완성 후 공개 권장 - 법적 면책 문구 설계를 초기부터 법률 검토와 병행 - "보고서의 정확성"에 대한 사용자 기대 관리 필요 (참고용 명시)
