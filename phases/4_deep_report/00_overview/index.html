    Phase 4: 부동산 딥 리포트 생성 시스템¶

    1. 배경 및 목적¶

    Phase 1~3에서 RAG 기반 Q&A 시스템을 구축하고 197개 질문 벤치마크로 평가한 결과, RAG 데이터의 유지보수 비용 대비 일반 Q&A에서의 효용이 제한적임을 확인했다. 특히:

    • 데이터 시의성 문제: 유튜브 전문가 콘텐츠 기반 RAG 데이터는 시간이 지나면 정책 변경, 시세 변동 등을 반영하지 못함
    • LLM 자체 지식과의 차이: 일반적인 부동산 질문에 대해 RAG 보강 답변과 LLM 단독 답변의 품질 차이가 기대보다 크지 않음
    • 유지보수 비용: 30,000+ 문서 변환/색인 파이프라인의 운영 비용이 높음

    이에 프로젝트의 원래 기획 방향인 "주소 하나를 입력하면 개별 물건에 대한 심층 분석 보고서를 생성하는 AI 어드바이저"로 회귀한다.

    핵심 전환¶

    Before: 일반 Q&A → RAG 검색 → LLM 답변
After:  주소 입력 → API 데이터 수집 → 정책 룰엔진 → LLM 종합 보고서

    RAG의 역할 변경¶

    • Before: 답변의 핵심 근거 (정량/정성 모두 RAG 의존)
    • After: 전문가 해석 레이어로만 선택적 사용 (정량 데이터는 API, 정책 계산은 룰엔진)

    목표¶

    1. 가이드형 인터뷰 UI: 자유 서술이 아닌, 단계별 보기 선택/입력 방식으로 사용자 조건 수집
    2. 아파트 주소 입력 시 7개 분석 축의 종합 딥 리포트 자동 생성 (일조/조망 분석 포함)
    3. 11개+ 공공/상용 API 연동으로 실시간 정량 데이터 기반 분석
    4. 정책 룰엔진으로 세금/대출 정확 계산 (LLM 의존 탈피)
    5. 거래 추이 그래프 (최근 3개월 거래량/금액 + 5년 가격 변동) 보고서 내 시각화
    6. 기존 RAG 인프라는 전문가 인사이트 보완용으로 유지
    7. 60초 이내 보고서 생성 (병렬 API + 병렬 LLM)
    8. 모든 설정값은 config/params.yaml로 외부화, 코드 전반에 기능 설명 주석 포함

    2. 경쟁사 분석 및 차별화¶

    2.2 경쟁사 공백 (Gap Analysis)¶

    1. 종합 딥리포트 부재: 어떤 서비스도 개별 물건에 대한 원스톱 종합 분석 보고서를 제공하지 않음
    2. 법률/규제 분석 자동화 전무: 용도지역, 건폐율/용적률, 행위 제한, 정비구역 자동 분석 없음
    3. 정책 연동 투자 시뮬레이션 없음: LTV/DSR + 세금 계산을 물건 분석에 통합한 서비스 없음
    4. RAG 기반 전문가 인사이트 없음: 전문가 의견을 검색하여 AI 근거로 제시하는 서비스 없음
    5. 미래 개발 계획 통합 분석 없음: 도시/교통계획을 특정 물건과 연결하는 서비스 없음

    2.4 핵심 차별화 5가지¶

    1. 원스톱 종합 리포트: 6개 분석 축을 하나의 보고서로 통합 (경쟁사 전무)
    2. 정책 룰엔진 + RAG 이중 구조: 세금/대출 정확 계산(룰엔진) + 전문가 해석 인용(RAG)
    3. 대화형 인터뷰 → 맞춤 보고서: 사용자 상황(투자/실거주/갈아타기)에 따라 세율/한도/강조점 변경
    4. RAG 전문가 인사이트: 11,000+ 유튜브 전문가 콘텐츠 기반 근거 제시
    5. 개별 물건 딥다이브: 탐색이 아닌 특정 물건 심층 분석 (Blue Ocean)

    3. 가이드형 인터뷰 UI¶

    3.1 설계 원칙¶

    일반 AI 채팅과 달리, 사용자의 자유 서술을 받지 않고 보고서 생성에 필요한 정보를 단계별로 선택/입력하도록 유도한다. Claude Code의 선택형 프롬프트와 유사하게, 각 단계에서 보기를 제시하고 사용자가 선택하거나 직접 입력할 수 있다.

    3.2 인터뷰 플로우¶

    [Step 1] 주소 입력
  AI: "분석할 부동산의 주소 또는 단지명을 입력해주세요."
      "(예: 마포래미안푸르지오, 서울시 강남구 역삼동 123-4, 마래푸)"
  사용자: "마포구 아현동 마포래미안푸르지오 101동 1502호" (자유 입력)
      │
      ▼
[Step 2] 주소 확인 (지도 표시)
  AI: "다음 주소가 맞으신가요?"
      "📍 서울시 마포구 마포대로 217 마포래미안푸르지오"
      "   101동 1502호 | 전용 84.82m² | 15층 | 2014년 준공"
      [지도 이미지 표시]
  보기: [1] 맞습니다 → Step 3으로
        [2] 다른 결과 보기 → 후보 목록 표시
        [3] 다시 입력하기 → Step 1로

  * 호수 미입력 시: 단지 전체 기준으로 분석 진행
  * 층수/면적: 건축물대장 API에서 자동 조회
      │
      ▼
[Step 3] 분석 목적 선택
  AI: "어떤 목적으로 분석하시나요?"
  보기: [1] 매매 — 구매 (실거주)

    설계 근거: - 주택수: 1주택 기준으로 기본 분석. 다주택 시나리오는 보고서 투자 수익률 섹션에 참고용으로 별도 표기 (2주택/3주택+ 세율 비교표) - 층수/면적: 호수 입력 시 건축물대장 API에서 자동 조회, 미입력 시 단지 평균/대표 면적 기준 - 인터뷰 4단계로 축소: 불필요한 질문 제거하여 보고서 생성까지 최소 인터랙션

    3.3 UI 구현 방식¶

    • Chainlit (MVP): cl.AskActionMessage로 버튼형 보기 제시, cl.AskUserMessage로 텍스트 입력
    • Next.js (Phase 4.3): 커스텀 스텝 컴포넌트 (라디오 버튼, 드롭다운, 지도 위젯)
    • 인터뷰 플로우는 config/params.yaml에서 단계별 보기 항목을 설정 가능

    3.4 UserContext 산출물¶

    인터뷰 완료 시 생성되는 사용자 컨텍스트:

    @dataclass
class UserContext:
    purpose: str           # "매매_실거주", "매매_투자", "매도", "전세", "경매", "기타"
    custom_notes: str      # 기타 선택 시 사용자 입력
    • 주택수: 1주택 기본. 보고서 투자 섹션에 다주택 시나리오 비교표 자동 포함
    • 층수/면적: 주소에서 자동 파악 (NormalizedAddress에 포함)
    • 세금 계산: purpose에 따라 취득세(매매) vs 전세보증금 분석 등 보고서 강조점 변경
    • 대출 한도: purpose에 따라 LTV/DSR 규제 차등 적용 (실거주 vs 투자)

    3.3 에러 처리 전략 (Graceful Degradation)¶

    Tier 실패 시 영향 대상 API

    Tier 3 (optional) 해당 하위 항목 생략 학교알리미, SGIS, 뉴스

    분석 항목 데이터 소스 산출물

    양도소득세 시나리오 정책 룰엔진 보유기간/거주기간/다주택 여부별 세율

    섹션 5: 일조/조망 분석 (신규)¶

    분석 항목 데이터 소스 산출물
    건물 향 분석 VWorld WFS (건축물 폴리곤) 주동 배치 방향 (남향/동향 등), 방위각
    일조량 추정 suncalc (Python) + 건축물대장 동지/하지 기준 직사광선 수광 시간 추정
    인접 건물 차폐 VWorld WFS + 건축물대장 전면 건물 높이/거리 → 그림자 차폐 시간 계산
    조망 분석 건축물대장 (층수) + Kakao Places 층수 순위, 주변 공원/하천 방향, 개방감
    위성사진 분석 VWorld WMS (항공사진) + Claude Vision 위성 이미지 기반 LLM 정성 분석
    조망 점수 종합 계산 0~100점 (층수 30% + 개방감 25% + 인접거리 25% + 향 20%)
    구현 방식 (3D 시뮬레이션 없이)¶

    호갱노노의 3D ray-tracing 수준은 아니지만, 공공 데이터 + 간단한 계산 + AI 이미지 분석으로 실용적 수준의 분석 제공.

    Step 1: 건물 향 도출 

    VWorld WFS → 건축물 폴리곤 → Shapely 최소외접사각형 → 장축 법선 방위각
→ "남남동향 (방위각 약 160도)"

    Step 2: 일조량 계산 

    suncalc → 동지(12/21) 시간대별 태양 방위각/고도
건물 향 법선 vs 태양 방위각 각도차 < 90° → 직사광선 수광
인접 건물 높이(건축물대장) / 이격거리(VWorld) → 차폐 고도 계산
→ "동지 기준 직사광선 약 5.8시간 (건축법 4시간 기준 충족)"

    Step 3: 위성사진 AI 분석 

    VWorld WMS → 항공사진(PNG) → Claude Vision
→ "남측으로 넓은 도로와 공원 접근성이 양호하며..."

    한계 vs 호갱노노¶
    항목 호갱노노 3D 우리 방식 (MVP)
    세대별 정밀도 높음 (창문 단위 ray-trace) 보통 (동 단위 외벽 방향)
    시각적 표현 3D 인터랙티브 모델 위성사진 + 텍스트 보고서
    그림자 애니메이션 시간 슬라이더 동지/하지 정적 추정
    구현 난이도 3~6개월 (팀) 10~12일
    AI 해석 없음 (사용자가 직접 판단) 자연어 해석 제공 (차별점)
    가격 연계 없음 남향 프리미엄 분석 연계
    자원/비용 분석¶
    항목 비용 일일 한도 비고
    VWorld WFS (건축물 폴리곤) 무료 40,000건/일 상업용 시 별도 승인 (무료 유지)
    VWorld WMS (항공사진) 무료 WFS와 공유 25~50cm 해상도, 1~2년 전 촬영분
    suncalc (태양 위치 계산) 무료 없음 순수 Python 계산, API 호출 없음
    Shapely (폴리곤 연산) 무료 없음 로컬 CPU, 단지 1개 < 0.1초
    Claude Vision (항공사진 분석) ~$0.02/건 API 한도 Sonnet 1024x1024 ≈ 1,600 토큰

    유료 API 불필요. 전체 파이프라인이 VWorld(무료) + 로컬 계산(suncalc/Shapely)으로 구성. 유일한 비용은 Claude Vision 항공사진 분석(건당 ~$0.02)으로, 보고서당 전체 LLM 비용 $0.10~0.30의 10% 이하.

    컴퓨팅 자원: Shapely 폴리곤 연산 + suncalc 태양 계산 모두 밀리초 단위. 특별한 GPU/메모리 불필요.

    참고: VWorld 항공사진은 1~2년 전 촬영분으로, 최근 신축/철거 건물은 미반영 가능. 부동산 분석 목적에서는 충분.

    필요 의존성¶
    • suncalc (Python, ~0.1MB): 태양 위치(방위각/고도) 계산
    • shapely (Python, GIS 라이브러리): 폴리곤 기하 연산
    • VWorld WFS API: 건축물 폴리곤 + 항공사진
    • 건축물대장 API: 층수 (grndFlrCnt)

    6.1.1 LLM 백엔드 전환 설계¶

    개발 단계에서는 Claude Code CLI를 사용하되, 프로덕션 전환 시 Anthropic SDK로 전환 가능하도록 설계한다.

    # config/params.yaml
llm:
  backend: "cli"                    # "cli" 또는 "api" — 한 줄 변경으로 전환
  cli:
    binary: "/usr/local/bin/claude-code"
    timeout: 180
    cooldown: 7                     # rate-limit 방지 쿨다운 (초)
    model: "sonnet"                 # claude-code --model 인자
  api:
    model: "claude-sonnet-4-6"
    max_retries: 3
    temperature: 0.0

    코드에서는 LLMClient 래퍼가 backend 설정에 따라 자동으로 CLI 또는 SDK를 선택하므로, 호출부 코드 변경 없이 params.yaml 한 줄 수정으로 전환 가능.

    6.1.2 설정 외부화 원칙¶

    모든 사용자 변경 가능 값은 config/params.yaml에 집중:

    카테고리 설정 항목 예시 변경 빈도
    API 키 카카오, data.go.kr, KOSIS 등 초기 1회
    LLM 설정 백엔드(cli/api), 모델명, 온도, 타임아웃 개발 중 수시
    캐시 TTL 실거래가 24h, 건축물 7d 등 드물게
    보고서 설정 섹션 활성화, 검색 반경, 거래 조회 개월수 드물게
    인터뷰 보기 목적/유형/층수 선택지 텍스트 드물게
    차트 스타일 색상, 폰트, 범례 위치 등 드물게
    면책 문구 보고서 하단 고지 문구 드물게

    6.1.3 코드 주석 원칙¶

    • params.yaml 각 항목에 한글 인라인 주석 (# 설명)
    • 모든 Python 클래스에 docstring (클래스 역할, 사용법)
    • 모든 public 함수에 docstring (파라미터, 반환값, 동작 설명)
    • 복잡한 로직에 인라인 주석 (왜 이렇게 구현했는지)
    • 모듈 파일 상단에 모듈 목적 docstring

    [1] 슬랭 확장: ontology slang_alias_mapping.json으로 약칭 해석
[2] Kakao Geocoding: 도로명/지번주소, 법정동코드(10자리), lat/lng
[3] PNU 생성: 법정동코드(10) + 산여부(1) + 본번(4) + 부번(4) = 19자리
[4] 단지명 추출: Kakao place_name 또는 사용자 입력 파싱

    최종 산출물 NormalizedAddress: 

    class NormalizedAddress(BaseModel):
    raw_input: str                    # 원본 입력
    sido_code: str                    # 2자리 (11=서울)
    sigungu_code: str                 # 5자리 (11680=강남구)
    bjdong_code_5: str                # 5자리 (실거래가 API용)
    bjdong_code_10: str               # 10자리 (전체 법정동코드)
    pnu: str                          # 19자리 (토지/건축물 API용)
    road_address: str                 # 도로명주소
    jibun_address: str                # 지번주소
    lat: float                        # 위도
    lng: float                        # 경도
    danji_name: str | None            # 아파트 단지명 (실거래가 매칭용)
    property_type: str                # apartment, house, land, commercial

    8. 구현 로드맵¶

    Phase 4.0: 기반 구축 (2주)¶

    Week 1: 핵심 인프라 - codes/generation/llm_client.py — LLM 클라이언트 (CLI/API 전환 가능, params.yaml 설정) - codes/api/base.py — BaseAPIClient (retry, rate-limit, cache, XML/JSON) - codes/api/cache.py — Redis CacheBackend + InMemory fallback - codes/api/rate_limiter.py — AsyncTokenBucket + DailyQuotaTracker - codes/api/clients/kakao.py — Kakao Geocoding + Place + Direction - codes/report/address.py — AddressNormalizationPipeline - codes/report/config/params.yaml — 전체 설정 파일 (API 키, LLM 설정, 캐시 TTL 등) - docker/docker-compose.yml — Redis + PostgreSQL 추가

    Week 2: 핵심 데이터 API - codes/api/clients/real_transaction.py — 실거래가 (12개월 병렬 fetch) - codes/api/clients/building_register.py — 건축물대장 - codes/api/clients/land_use.py — 토지이용규제 - codes/api/clients/vworld.py — VWorld 필지경계/토지이용계획/항공사진 - Pydantic 모델: Transaction, BuildingInfo, LandUseInfo

    Phase 4.1: MVP 리포트 (4주)¶

    Week 3: 가이드형 인터뷰 + 리포트 엔진 코어 - codes/report/interview.py — 가이드형 인터뷰 플로우 (Chainlit cl.AskActionMessage) - ReportState (UserContext 포함), ReportOrchestrator, ParallelDataCollector - 가격/시세 분석 섹션 (프롬프트 + LLM 합성기)

    Week 4: 거래 차트 + 입지 + 법률 섹션 - codes/report/charts.py — matplotlib 차트 생성 (5년 추이, 3개월 거래량, 듀얼축 콤보) - 학교알리미, SGIS, KOSIS API 클라이언트 - 입지 분석 / 법률/규제 분석 섹션 - RAG 통합 (선택적 SearchPipeline 호출)

    Week 5: 일조/조망 분석 - codes/report/sunview.py — 일조/조망 분석 엔진 - VWorld WFS 건축물 폴리곤 → Shapely 향 도출 - suncalc 일조량 계산 (동지/하지) - 인접 건물 차폐 계산 - VWorld WMS 항공사진 → Claude Vision 정성 분석 - 조망 점수 계산 (0~100)

    Week 6: UI 통합 + E2E 테스트 - Chainlit 리포트 모드 핸들러 (인터뷰 → 보고서 생성 → 차트/이미지 포함 출력) - Q&A/리포트 모드 자동 분기 - 보고서 조립기 + 요약 결론 - 10개 골든 주소 테스트 (서울 5, 수도권 3, 지방 2)

    MVP 성공 기준: - 아파트 주소 정규화 성공률 90%+ - 보고서 생성 시간 90초 이내 (일조 분석 포함) - 5개 섹션 (가격/입지/법률/일조조망/리스크) 실제 API 데이터 기반 생성 - 가격 차트 4종 (5년 추이, 3개월 거래, 주변 비교, 듀얼축) 포함 - 인터뷰 기반 사용자 조건 반영 - Markdown 구조화 보고서 출력 - params.yaml 설정만으로 동작 커스터마이징 가능

    Phase 4.2: 완전 리포트 (4주)¶

    Week 7-8: 정책 룰 엔진 + 투자 섹션 - 세율 테이블 (YAML), 취득세/양도세/보유세 계산기, LTV/DSR 계산기 - 투자 수익률 섹션 (인터뷰 UserContext 연동 — 주택수/목적에 따라 세율 변경)

    Week 9: 미래가치 섹션 + 추가 API - 네이버 뉴스, 빅카인즈, 온비드 API 클라이언트 - 미래 가치 섹션

    Week 10: PDF + 안정화 - PDF 보고서 출력 (차트/이미지 포함) - 30개 골든 주소 테스트 (7개 섹션 전체)

    Phase 4.3: 프리미엄 (4주+)¶

    10. 리스크 및 대응¶

    리스크 발생 확률 영향 대응
    data.go.kr API 키 승인 지연 (1-3영업일) 중 MVP 지연 Phase 4.0 첫날 전체 신청, Mock 데이터로 개발
    개발 계정 일일 1,000건 제한 중 개발 속도 적극적 캐싱 + 운영 계정 전환 신청
    비아파트 주소 정규화 실패 높음 커버리지 MVP는 아파트 전용, Phase 4.2에서 확장
    세금 계산 정확도 높음 신뢰도 "참고용 시뮬레이션" 면책 고지, 세무사 상담 권장
    LLM 할루시네이션 중 품질 API 데이터 블록 명시 분리, 숫자 생성 금지 프롬프트
    보고서 생성 시간 > 2분 중 UX 병렬 API + 병렬 LLM, 스트리밍 진행 상태 표시
    Anthropic API 비용 증가 낮음 (MVP) 비즈니스 보고서당 ~$0.10-0.30 (Sonnet), 캐싱으로 중복 절감
