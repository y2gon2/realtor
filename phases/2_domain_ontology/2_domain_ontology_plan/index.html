    Phase 2: 도메인 온톨로지 구축 계획¶

    진입 시점: Phase 1 임베딩 품질 평가 완료 직후 목표: 구어체/일상 표현 → 전문용어 매핑을 위한 부동산 도메인 온톨로지 구축 및 용어 검색 파이프라인 연동

    0. 문제 정의 및 배경¶

    0-1. 발견된 문제¶

    Phase 1 임베딩 품질 평가(20260315_4_embedding_quality_test)에서 다음 문제가 확인되었다:

    사용자 구어체 질의 → 임베딩 → 전문용어 기반 문서와 의미 공간 불일치 → 다른 문서 검색
    메트릭 결과 기준 판정
    유사 쌍 Overlap@5 20% > 40% 미달
    유사 쌍 Overlap@10 26% > 30% 미달 (근접)

    핵심 원인: 문서 부족이 아니라, 구어체 질의가 전문용어의 임베딩 의미 공간을 벗어나 다른 문서 집합을 검색하는 것이 원인이다.

    질의 A (전문용어) 질의 B (구어체) 키워드↔B 갭 Overlap@10
    DSR 규제가 대출한도에 미치는 영향 총부채원리금상환비율 때문에 대출이 줄어드나요? 0.31 10%
    재건축 초과이익환수제 설명 재건축 부담금이 뭔가요? 0.30 20%
    다주택자 세금 중과 집 여러 채 가진 사람 세금 불이익 0.23 20%

    키워드↔질의B 유사도 갭이 클수록 Overlap이 낮아지는 강한 상관관계가 확인됨.

    0-2. 해결 전략: 도메인 온톨로지 기반 2단계 검색¶

    검토한 대안과 선택 근거:

    방안 설명 판단
    LLM 질의 정규화 (프롬프트 내 용어 사전) 매 질의마다 LLM 호출 + 용어 사전을 프롬프트에 포함 용어 수 한계, 매 질의 input 토큰 비용
    LLM 질의 정규화 (few-shot만) LLM의 기존 지식에 의존 도메인 특화 용어 커버리지 불확실
    도메인 온톨로지 임베딩 DB (채택) 별도 용어 DB 구축 → 벡터 검색으로 용어 매핑 확장성, 비용 효율, 재사용성 우수
    BM25 기반 용어 검색 키워드 토큰 매칭으로 용어 검색 유사 변형 표현 커버 불가

    0-3. 도메인 온톨로지를 독립 Phase로 분리하는 이유¶

    부동산 정보 데이터와 도메인 개념 데이터는 근본적으로 성격이 다르다:

    속성 부동산 정보 데이터 (Phase 1) 도메인 개념 데이터 (Phase 2)
    변동성 높음 — 시세, 정책, 뉴스가 계속 변함 낮음 — 용어 체계 자체는 거의 안 변함
    확장 방식 동일 파이프라인에 데이터만 추가 구조 설계가 핵심, 나중에 구조 변경 시 전면 재작업
    MVP 접근 가능 — 데이터만 늘리면 됨 위험 — 대충 구축하면 이후 완전 재작업 필요
    수명 데이터 단위로 갱신 한 번 구축하면 장기간 재사용

    법이 바뀌어도 "양도소득세", "취득세", "종합부동산세"라는 개념 체계 자체는 유지된다. 세부 세율이나 조건만 달라질 뿐이다. 따라서 온톨로지는 지금 제대로 구축하는 것이 장기적으로 가장 효율적이다.

    1. 2단계 검색 파이프라인 설계¶

    1-1. 전체 아키텍처¶

    [사용자 질의]
"집 여러 채 가진 사람 세금 불이익"
    │
    ▼
┌──────────────────────────────────────┐
│ Stage 1: 용어 임베딩 DB 검색          │
│                                      │
│  질의 임베딩 → 용어 DB 코사인 검색     │
│  top-3 매칭 용어 반환:                │
│   ├── 다주택자 양도소득세 중과          │
│   ├── 다주택자 취득세 중과             │
│   └── 다주택자 종합부동산세 중과        │
└──────────────┬───────────────────────┘
               │
               ▼
┌──────────────────────────────────────┐
│ Stage 2: 기존 RAG 임베딩 DB 검색      │
│                                      │
│  검색 A: 원본 질의 → top-K            │
│  검색 B: 매칭 용어 → top-K (각각)      │
│                                      │
│  RRF 합산 → 최종 top-K 반환           │
└──────────────────────────────────────┘
    │
    ▼
[LLM 답변 생성]

    1-2. Stage 1 상세 — 용어 매칭¶

    입력: 사용자 질의 (구어체)
    ↓ KURE-v1 임베딩 (기존 모델 공유)
    ↓ 용어 DB 코사인 유사도 검색 (top-3)
    ↓ score threshold 적용 (예: > 0.65)
출력: 매칭된 전문용어 리스트

    score threshold가 필요한 이유: "오늘 날씨 어때?"처럼 부동산과 무관한 질의에서 용어가 잘못 매칭되는 것을 방지한다. threshold 미달 시 Stage 1을 건너뛰고 원본 질의만으로 Stage 2를 실행한다.

    1-3. Stage 2 상세 — 다중 질의 검색 + RRF 합산¶

    원본 질의: "집 여러 채 가진 사람 세금 불이익" → 검색 → 결과 A
매칭 용어 1: "다주택자 양도소득세 중과"       → 검색 → 결과 B₁
매칭 용어 2: "다주택자 취득세 중과"           → 검색 → 결과 B₂
매칭 용어 3: "다주택자 종합부동산세 중과"      → 검색 → 결과 B₃
    ↓
RRF(결과 A, B₁, B₂, B₃)  →  최종 top-K

    RRF (Reciprocal Rank Fusion) 공식:

    RRF_score(doc) = Σ  1 / (k + rank_i(doc))
                 i

k = 60 (표준 상수)
rank_i(doc) = 검색 결과 i에서 해당 문서의 순위 (없으면 무한대)

    1-4. 기대 효과¶

    현재 (Stage 1 없음) 개선 후 (2단계 검색)
    "집 여러 채" → 임베딩이 "다주택자" 문서와 거리 멀어 누락 "집 여러 채" → 용어 DB에서 "다주택자" 발견 → 정확한 문서 검색
    키워드↔질의B 갭 0.23~0.31 매칭 용어로 검색하므로 갭 사실상 0
    Overlap@10: 10~20% 원본+용어 RRF로 대폭 개선 예상

    1-5. 비용 분석¶

    항목 비용
    Stage 1 용어 검색 벡터 검색 1회 (~1ms), 1,000~2,000 포인트 대상 — 거의 무비용
    Stage 2 추가 검색 벡터 검색 1~3회 추가 (~3ms) — 무시 가능
    LLM 호출 추가 없음 — LLM은 최종 답변 생성 1회만
    인프라 기존 Qdrant에 컬렉션 1개 추가 — 추가 서버 불필요

    2-3. 임베딩 텍스트 생성 규칙¶

    각 엔트리에서 1개의 임베딩 벡터를 생성한다. 임베딩 입력 텍스트는 다음과 같이 조합한다:

    임베딩 입력 = term + " | " + aliases (전부 줄바꿈 연결) + " | " + description

    예시:

    다주택자 양도소득세 중과
집 여러 채 팔 때 세금 더 내는 것
다주택 양도세 중과세율
2주택 3주택 양도세 불이익
다주택자 양도세 페널티
집 많으면 팔 때 세금 폭탄
다주택자 양도세 중과 적용
2주택 이상 보유자가 조정대상지역 내 주택을 양도할 때 기본세율에 20~30%p를 가산하여 부과하는 제도. 주택 수, 보유 기간, 지역에 따라 중과세율이 달라진다.

    이 방식의 핵심: aliases에 구어체 표현이 포함되어 있으므로, 구어체 질의와 이 임베딩 벡터 간의 코사인 유사도가 높아진다. term만 임베딩하면 원래 문제(구어체↔전문용어 거리)를 그대로 답습하지만, aliases를 풍부하게 포함함으로써 이를 해결한다.

    2-4. 규모 추정¶

    항목 수량
    Level 1 (대분류) ~15개
    Level 2 (기본 개념) ~150-200개
    Level 3 (세부 적용) ~800-1,500개
    총 엔트리 ~1,000-1,700개
    엔트리당 aliases 5-10개
    총 aliases 작성량 ~5,000-17,000개
    임베딩 벡터 수 엔트리당 1개 = ~1,000-1,700개
    Qdrant 추가 메모리 ~10MB 미만 (기존 93,943 포인트 대비 미미)

    벡터 수 vs aliases 수

    aliases가 5,000~17,000개라고 해서 벡터가 그만큼 필요한 것이 아니다. 한 엔트리의 모든 aliases는 하나의 텍스트로 합쳐져 벡터 1개로 임베딩된다. aliases는 해당 벡터가 구어체 표현 근처에도 위치하도록 만드는 역할이다.

    3. 온톨로지 구축 방법론¶

    3-1. 전체 구축 프로세스¶

    Phase A: 구조 설계 (사람)
│  ├── Level 1 대분류 체계 확정
│  ├── Level 2 기본 개념 목록 작성
│  ├── 스키마 확정 + 샘플 30~50개 직접 작성
│  └── 샘플로 임베딩 검색 테스트 → 스키마 검증
│
Phase B: 대량 생성 (LLM 배치)
│  ├── Level 2 목록 + 구조 규칙을 LLM에 제공
│  ├── Level 3 세부 적용 엔트리 자동 생성
│  ├── 각 엔트리의 aliases 5-10개 자동 생성
│  └── description 자동 생성
│
Phase C: 검수 및 보완 (사람)
│  ├── 전문성 필요 영역 위주 검수 (세금, 대출 등)
│  ├── 누락된 조합/표현 보완
│  ├── 중복 엔트리 정리
│  └── related_terms 연결 검증
│
Phase D: 임베딩 + 통합 테스트
    ├── 전체 엔트리 임베딩 → Qdrant 용어 컬렉션 생성
    ├── 기존 테스트 8쌍으로 Overlap 개선 측정
    ├── 추가 구어체 테스트 케이스로 검증
    └── threshold 및 top-K 파라미터 튜닝

    • 임베딩 후 기존 테스트 질의(구어체)와 코사인 유사도 측정 → 0.7 이상이면 스키마 유효

    3-3. Phase B: LLM 배치 생성 — 상세¶

    프롬프트 설계¶

    [시스템]
당신은 대한민국 부동산 도메인 전문가입니다.
아래 규칙에 따라 부동산 용어 온톨로지 엔트리를 생성하세요.

[규칙]
1. 주어진 Level 2 용어에 대해 실제 사용되는 Level 3 세부 적용을 모두 나열하세요.
2. 각 엔트리의 aliases는 5-10개로, 반드시 다음을 포함:
   - 공식 명칭 변형 (약어, 풀네임)
   - 구어체/일상 표현 (일반인이 실제 사용하는 표현) 2개 이상
   - 검색 시 사용할 수 있는 키워드 조합
3. description은 1-2문장으로 제도/개념의 핵심을 설명하세요.
4. related_terms는 함께 알아야 할 관련 개념의 id를 나열하세요.

[Level 2 용어]
category_path: "세금 > 양도소득세"
parent_id: "tax_transfer"
term: "양도소득세"

[샘플 출력 (참고)]
{샘플 엔트리 3-5개}

[요청]
위 Level 2의 모든 Level 3 엔트리를 JSON 배열로 생성하세요.

    배치 실행 계획¶

    단계 작업 LLM 예상 호출 수

    aliases 보강 생성된 Level 3의 aliases 품질 검증 및 추가 Claude Sonnet ~175회
    description 검증 전문 용어 정의의 정확성 확인 Claude Sonnet ~50회 (샘플링)

    3-4. Phase C: 검수 — 상세¶

    검수 우선순위¶

    우선순위 대분류 이유
    P1 세금, 대출/금융 수치와 조건이 정확해야 함. 오류 시 실질적 피해 가능
    P1 규제/정책 정책 변경이 잦아 최신성 확인 필요
    P2 경매/공매, 등기/권리 법률 용어가 많아 aliases 정확성 중요
    P3 나머지 일반적 부동산 상식 수준

    검수 체크리스트¶

    • [ ] 동일 개념이 서로 다른 엔트리로 중복 생성되지 않았는가
    • [ ] aliases에 구어체 표현이 2개 이상 포함되어 있는가
    • [ ] related_terms의 id가 실제 존재하는 엔트리를 가리키는가
    • [ ] description이 특정 시점의 세율/조건이 아닌 제도 자체를 설명하는가
    • [ ] category_path와 parent_id가 계층 구조와 일치하는가

    테스트 계획¶

    테스트 1: 기존 8쌍 재검증

    임베딩 품질 평가의 8개 유사 쌍에 대해 2단계 검색 적용 후 Overlap 재측정:

    목표 기준
    Overlap@5 40% 이상 (현재 20%)
    Overlap@10 50% 이상 (현재 26%)

    테스트 2: 용어 매칭 정밀도

    구어체 질의 50개에 대해 용어 DB 검색 결과의 정확성 측정:

    메트릭 목표
    Precision@3 70% 이상 (top-3 중 관련 용어 비율)
    무관 질의 필터링 threshold 미달 정확도 90% 이상

    테스트 3: 엔드-투-엔드 RAG 품질

    2단계 검색 파이프라인 전체를 연결하여 최종 답변 품질 측정.

    4. 주요 설계 결정 및 리스크¶

    4-1. 설계 결정¶

    결정 사항 선택 근거
    용어 DB 검색 방식 Dense (코사인 유사도) aliases를 풍부하게 포함하여 구어체 커버, BM25는 유사 변형 표현 누락 위험
    임베딩 모델 KURE-v1 (기존과 동일) 모델 통일로 운영 복잡도 최소화, 추가 모델 로드 불필요
    엔트리당 벡터 수 1개 (aliases 합산) aliases가 많아져도 벡터 수 = 엔트리 수로 관리 가능
    결과 합산 RRF 이질적인 검색 결과 합산에 가장 안정적인 방법
    용어 DB 저장소 기존 Qdrant에 컬렉션 추가 추가 인프라 불필요

    4-2. 리스크 및 대응¶

    리스크 영향 대응
    용어 매칭 노이즈 — "부담금"으로 검색 시 "개발부담금", "기반시설부담금" 등 다수 매칭 Stage 2 검색 결과 오염 score threshold 적용, top-K를 2~3으로 제한, 테스트로 최적값 튜닝
    aliases 품질 부족 — LLM이 생성한 구어체 표현이 실제 사용자 표현과 괴리 매칭률 저하 Phase C 검수 + 운영 후 사용자 질의 로그 분석으로 aliases 지속 보강
    Level 3 조합 폭발 — "다주택자" × "세금 종류" × "지역/조건"으로 엔트리 과다 구축 비용 증가, 유지보수 부담 Level 3은 실제 검색 빈도가 높은 조합 위주로 선별. "다주택자 + 조정대상지역 + 양도세 + 장기보유특별공제" 수준의 4중 조합은 제외
    임베딩 텍스트 길이 — aliases가 많으면 합산 텍스트가 길어져 임베딩 품질 희석 특정 alias와의 유사도 하락 KURE-v1은 8,192 토큰 지원. aliases 10개 + description은 200토큰 이내로 충분. 희석 여부는 Phase A 샘플 테스트에서 검증

    6. 향후 확장 방향¶

    6-1. 운영 단계 aliases 자동 보강¶

    사용자 질의 로그 수집
    ↓ 용어 DB 매칭 실패 (threshold 미달) 건 추출
    ↓ 수동/LLM 분석으로 해당 질의가 어떤 용어에 매핑되어야 하는지 판단

    6-2. Reranker 연계¶

    Stage 2 검색 결과에 Cross-Encoder Reranker를 추가하여 최종 순위 정밀도를 높일 수 있다. 이는 Phase 1 리포트 7-3에서 제안된 방안과 동일하며, 온톨로지 기반 검색과 독립적으로 적용 가능하다.

    6-3. 타 도메인 확장¶

    온톨로지 스키마와 파이프라인은 도메인 독립적으로 설계되었다. 부동산 외 다른 도메인(금융, 법률, 의료 등)에도 동일한 구조로 확장할 수 있다.
