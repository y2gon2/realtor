    2-3. 쌍별 상세 분석¶

    #1 전세사기 예방 — Overlap 최우수 (60%/50%)¶

    • 키워드 문서풀: 15개 고유 doc, 전부 score > 0.7로 좁지만 집중된 문서풀
    • 질의A: 키워드와 cos=0.9267, top-10 전부(8개 고유 doc) 키워드 풀 안에 포함
    • 질의B: 키워드와 cos=0.8368로 여전히 높음. 풀 밖 문서는 3개뿐
    • Overlap 높은 이유: "전세사기"라는 구체적 주제 → 관련 문서 풀이 좁아서 A, B 모두 같은 소수의 핵심 문서를 상위에 올림

    #2 강남 아파트 시세 — Overlap 양호 (20%/40%)¶

    • 키워드 문서풀: 17개 고유 doc이지만 score > 0.7은 2개뿐으로 넓고 산만한 문서풀
    • 질의A: "시세 전망"이라는 추가 의도가 키워드 풀 밖 6개 문서를 끌어옴
    • 질의B: "가격이 앞으로 어떻게 될까요?"라는 전망 표현이 키워드 풀 밖 8개 문서를 끌어옴
    • Overlap 중간인 이유: A, B 모두 키워드 풀 밖에서 검색하지만, "전망/가격 예측" 의도가 공유되어 일부 문서가 겹침 (top-10에서 4개)

    #3 재건축 초과이익환수제 — Overlap 저조 (0%/20%)¶

    • 키워드 문서풀: 16개 고유 doc, score > 0.7이 16개로 충분한 문서풀
    • 질의A: 키워드와 cos=0.9267, top-10 전부 키워드 풀 안 (0개 밖)
    • 질의B: 키워드와 cos=0.6298, top-10 중 6개가 키워드 풀 밖
    • Overlap 낮은 이유: "초과이익환수제"(전문 용어) vs "부담금"(일상 용어) 간 임베딩 거리가 큼(갭 0.2969). 질의B가 "부담금"이라는 광의적 표현에 매칭되는 다른 주제의 문서(재건축 부담금이 아닌 일반 부동산 부담금 관련)를 찾아옴. 문서 부족이 아닌, 질의B의 의미 이탈이 원인.

    #4 DSR 대출한도 — Overlap 최저 (0%/10%)¶

    • 키워드 문서풀: 20개 고유 doc, 전부 score > 0.7으로 풍부한 문서풀
    • 질의A: 키워드와 cos=0.8606, 8개가 키워드 풀 안
    • 질의B: 키워드와 cos=0.5492 (최저), top-10 중 8개가 키워드 풀 밖
    • Overlap 낮은 이유: "DSR" 약어 vs "총부채원리금상환비율" 풀네임 간 임베딩 거리 최대(갭 0.3114). 질의B는 키워드 문서풀과 거의 무관한 문서를 검색함. 키워드 문서는 20개로 충분하나, 질의B가 완전히 다른 의미 공간에서 검색하는 것이 원인.

    #5 생애최초 주택구입 — Overlap 저조 (20%/10%)¶

    • 키워드 문서풀: 19개 고유 doc이지만 score > 0.7은 1개뿐으로 산만한 문서풀
    • 질의A: 키워드와 cos=0.8184이지만, top-10 중 7개가 키워드 풀 밖
    • 질의B: 키워드와 cos=0.6644, top-10 중 9개가 키워드 풀 밖
    • Overlap 낮은 이유: 이중 요인. (1) 키워드 자체의 문서풀 집중도가 낮음 (score > 0.7이 1개). (2) "처음 집 살 때 받을 수 있는 지원"이라는 일상 표현이 주택구입뿐 아니라 내집마련, 청약, 정부지원 등 광범위한 주제로 발산. 키워드 문서풀의 약한 집중도 + 질의B의 의미 발산 복합 원인.

    #6 경매 투자 주의사항 — Overlap 중간 (20%/30%)¶

    • 키워드 문서풀: 20개 고유 doc, score > 0.7이 12개로 충분한 문서풀
    • 질의A: 키워드와 cos=0.9889 (거의 동일), top-10 전부 키워드 풀 안
    • 질의B: 키워드와 cos=0.7844, top-10 중 6개가 키워드 풀 밖
    • Overlap 중간인 이유: 질의A는 키워드와 거의 동일하여 핵심 문서를 정확히 검색. 질의B는 "법원 경매로 집 살 때 조심할 점"이라는 구어체가 일부 다른 주제(주택 매매 시 주의사항 등)를 끌어들이나, "경매" 키워드가 공유되어 30% overlap 유지.

    #7 다주택자 세금 중과 — Overlap 중간 (20%/20%)¶

    • 키워드 문서풀: 20개 고유 doc, 전부 score > 0.7으로 풍부한 문서풀
    • 질의A: 키워드와 cos=1.0000 (동일), top-10 전부 키워드 풀 안
    • 질의B: 키워드와 cos=0.7689, top-10 중 8개가 키워드 풀 밖
    • Overlap 낮은 이유: "집 여러 채 가진 사람 세금 불이익"이라는 완전 구어체가 "다주택자 세금 중과" 전문 표현과 의미 공간에서 분리됨(갭 0.2311). 키워드 문서는 20개 이상 충분하나, 질의B가 구어체로 인해 다른 문서(재산세, 종부세 일반론 등)를 찾아옴.

    #8 신축 아파트 청약 — Overlap 중간 (20%/30%)¶

    • 키워드 문서풀: 19개 고유 doc이나 score > 0.7은 0개로 가장 산만한 문서풀
    • 질의A: 키워드와 cos=0.8264이지만, top-10 중 9개가 키워드 풀 밖
    • 질의B: 키워드와 cos=0.6689, top-10 중 10개 전부 키워드 풀 밖
    • Overlap 중간인 이유: 키워드 문서풀 자체의 집중도가 가장 낮아(score > 0.7 = 0개), 키워드-질의A-질의B가 각각 다른 문서 집합을 검색. 그럼에도 A, B 간 3개 문서가 겹치는 것은 "청약/분양"이라는 상위 의미가 공유되기 때문. 키워드 문서풀의 낮은 집중도가 주 원인.

    6-3. 강점¶

    1. 판별력 우수 (0.3684): 기준치 0.15의 2.5배. 유사/비유사 질의 간 의미적 거리가 명확히 구분됨
    2. 비유사 쌍 완전 분리: Overlap@5/10 모두 0% — 서로 다른 주제의 질의가 완전히 다른 문서를 반환
    3. cos_sim 기준 충족: 유사 쌍 평균 0.7606 (> 0.7), 비유사 쌍 평균 0.3922 (< 0.5)
    4. chunk_type 설계 의도 부합: hyde가 자연어 질의에 가장 높은 점수로, 설계 의도대로 동작
    5. 데이터 커버리지 양호: 모든 chunk_type에서 empty_count = 0

    6-4. 약점 및 주의사항¶

    1. Overlap@5/10 미달: 유사 쌍의 실제 검색 결과 겹침이 기준보다 낮음 (Overlap@5: 20% vs 기준 40%, Overlap@10: 26% vs 기준 30%)
    2. 근본 원인 (키워드 문서풀 분석 결과): 문서 부족이 아님. 키워드 문서풀은 15~20개로 충분. 질의B(구어체/일상 표현)가 키워드 의미 공간을 벗어나 다른 문서 집합을 검색하는 것이 주된 원인. 키워드↔질의B 유사도 갭이 클수록 Overlap이 낮아지는 강한 상관관계 확인.
    3. 원인 유형별 분류:
      • 질의B 의미 이탈 (4쌍: #3, #4, #6, #7): 키워드 문서 충분하나 구어체가 다른 의미 공간 검색
      • 키워드 문서풀 집중도 부족 (2쌍: #5, #8): 주제 자체가 넓어 키워드 검색 결과가 산만
      • 복합 원인 (1쌍: #2): 넓은 주제 + "전망" 의도 추가

    4. 대응: (1) query expansion으로 구어체→전문용어 매핑, (2) reranker로 의미적 재정렬, (3) top-K 확대

    5. 전문 용어 vs 일상 표현 간 임베딩 거리: 키워드↔질의B 유사도 갭이 0.09~0.31까지 분포. 특히 약어/전문용어 쌍에서 갭이 큼

    6. "초과이익환수제" ↔ "부담금" — 갭 0.30, overlap@10: 20%
    7. "DSR" ↔ "총부채원리금상환비율" — 갭 0.31, overlap@10: 10%
    8. "다주택자 세금 중과" ↔ "집 여러 채 가진 사람 세금 불이익" — 갭 0.23, overlap@10: 20%
    9. 대응: query expansion, synonym mapping, 또는 LLM 기반 질의 정규화(query normalization)로 보완

    P1 LLM 기반 질의 정규화 (query normalization) 구어체 질의를 전문 용어 기반으로 변환 후 검색. Overlap 저조의 근본 원인인 "질의B 의미 이탈" 직접 해결
    P2 RAG top-K 설정 시 10 이상 권장 구어체 질의 시 핵심 문서가 top-5 밖에 위치할 수 있으므로, 충분한 top-K 확보 후 LLM이 종합 판단
    P2 hyde chunk_type 우선 검색 자연어 질의에 가장 높은 매칭 점수 — RAG 1차 검색에 적합
    P2 Reranker 도입 검토 의미적 재정렬로 구어체/전문용어 표현 차이에 의한 순위 편차 보완
    P3 전문 용어 query expansion / synonym mapping DSR → 총부채원리금상환비율, 부담금 → 초과이익환수제 등 약어·동의어 매핑으로 검색 커버리지 확대

    7. Overlap 저조 개선 방안 상세¶

    7-0. 문제 정의¶

    사용자 구어체 질의 → 임베딩 → 전문용어 기반 문서와 의미 공간 불일치 → 다른 문서 검색

    Overlap 저조의 근본 원인은 두 가지로 분류된다:

    원인 유형 해당 쌍 핵심 문제
    질의B 의미 이탈 #3, #4, #6, #7 구어체/일상 표현이 전문용어와 다른 의미 공간에 임베딩됨
    키워드 문서풀 집중도 부족 #5, #8 주제 자체가 넓어 키워드 검색 결과가 산만

    아래 4가지 방안을 효과/실현성 기준으로 분석한다.

    7-1. 질의 전 LLM 전처리 (Query Normalization)¶

    키워드↔질의B 갭 자체를 줄이는 가장 직접적인 해결책.

    필요한 데이터¶

    항목 상세 비고
    부동산 도메인 용어 사전 약어↔풀네임, 전문용어↔일상표현 매핑 테이블 LLM 프롬프트에 참조용으로 제공
    few-shot 예시 구어체→전문용어 변환 입출력 예시 5~10쌍 프롬프트 내 포함
    LLM 경량 모델 (Haiku급) — latency 최소화 RAG 파이프라인에 이미 LLM 사용 중

    용어 사전 예시¶

    # 약어 ↔ 풀네임
DSR ↔ 총부채원리금상환비율
LTV ↔ 주택담보대출비율 (담보인정비율)
DTI ↔ 총부채상환비율

# 전문용어 ↔ 일상표현
초과이익환수제 ↔ 재건축 부담금
중과세율 ↔ 세금 불이익, 세금 더 내는 것
다주택자 ↔ 집 여러 채 가진 사람
생애최초 주택구입 ↔ 처음 집 사는 것
경매 낙찰 ↔ 법원에서 집 사는 것
청약 ↔ 새 아파트 분양 신청

    프로세스¶

    사용자 질의: "집 여러 채 가진 사람 세금 불이익"
    ↓ LLM (경량 모델, 용어 사전 참조)
정규화 질의: "다주택자 양도소득세 취득세 종합부동산세 중과세율"
    ↓ 임베딩 → 검색

    예상 효과¶

    • 키워드↔질의B 갭 0.31 → 0.05~0.10 수준으로 축소 예상

    • 3, #4, #6, #7 (의미 이탈 유형)에 직접적 효과¶

    비용/리스크¶

    • 매 질의마다 LLM 호출 1회 추가 → latency + 비용 증가
    • 정규화 과정에서 원래 의도가 왜곡될 가능성 → few-shot 품질이 중요

    7-2. Multi-Query Retrieval (질의 확장 검색)¶

    질의를 정규화하는 대신, 여러 변형을 동시에 검색하여 결과를 합치는 방법. 방법 7-1과 통합하여 1회 LLM 호출로 정규화 + 변형을 동시에 생성할 수 있다.

    필요한 데이터¶

    항목 상세 비고
    LLM 변형 질의 2~3개 생성 7-1과 동일 호출에서 처리 가능
    알고리즘 결과 합산 방식 — RRF (Reciprocal Rank Fusion) 또는 max-score merge 구현 필요
    추가 데이터 불필요 (LLM이 자체적으로 변형 생성)

    프로세스¶

    원본 질의: "집 여러 채 가진 사람 세금 불이익"
    ↓ LLM (1회 호출)
정규화: "다주택자 세금 중과"
변형 1: "다주택자 양도소득세 중과세율"
변형 2: "2주택 3주택 취득세 종부세 중과"
    ↓ 각각 임베딩 → 검색 (top-K씩)
    ↓ 결과 합산 (RRF)
    ↓ 최종 top-K 반환

    예상 효과¶

    • 7-1보다 recall이 높음 — 원본 구어체가 잡는 문서 + 전문용어가 잡는 문서 모두 확보

    • 5, #8 (문서풀 산만 유형)에도 효과 — 다양한 변형이 넓은 문서풀을 커버¶

    비용/리스크¶

    • 벡터 검색 횟수 3~4배 증가 (단, 벡터 검색 자체는 ms 단위로 빠름)
    • LLM 호출은 7-1과 합쳐 1회로 처리 가능

    7-3. Reranker (Cross-Encoder 기반 재정렬)¶

    검색 후 단계에서 질의-문서 쌍의 관련성을 재평가. 방법 7-1, 7-2와 병행 사용 가능.

    필요한 데이터¶

    항목 상세 비고
    한국어 Cross-Encoder 모델 ko-reranker, bge-reranker-v2-m3 등 모델 선정 및 평가 필요
    GPU 추론 인프라 기존 embedding 서버에 추가 로드 또는 별도 서버 추론 비용 발생
    학습 데이터 pretrained 모델 사용 시 불필요. 부동산 도메인 파인튜닝 시 질의-문서 관련성 쌍 필요 선택사항

    프로세스¶

    질의 → 임베딩 검색 (top-20~50 확보)
    ↓
Cross-Encoder가 (질의, 문서) 쌍마다 관련성 점수 산출
    ↓
재정렬된 top-K 반환

    Overlap 문제 해결 원리¶

    • Bi-encoder(KURE-v1): 질의와 문서를 독립적으로 임베딩 → "부담금" ↔ "초과이익환수제" 같은 간접적 의미 관계를 놓침
    • Cross-encoder: 질의와 문서를 동시에 입력받아 상호 참조 → "재건축 부담금이 뭔가요?" 질의와 "초과이익환수제" 문서 내용을 함께 보고 관련성 판단 가능

    예상 효과¶

    • 7-1/7-2로 확보한 후보군의 순서를 최적화
    • 구어체 질의와 전문용어 문서 간 매칭 정확도 향상

    비용/리스크¶

    • 검색 범위를 넓혀야(top-20~50) 의미가 있음 → 초기 검색 비용 증가
    • Cross-encoder 추론 비용 (top-30 문서 × 추론 1회씩)

    7-4. 문서 측 보강 (인덱싱 단계)¶

    질의를 바꾸는 대신, 문서 쪽에 구어체 표현을 추가하는 방법.

    선택지¶

    방식 설명 필요 데이터
    구어체 HyDE 추가 기존 hyde 청크 외에, "일반인이 할 법한 질문"을 추가 생성하여 색인 LLM으로 문서당 2~3개 생성
    synonym 메타데이터 문서 내 전문용어에 대한 일상 표현을 메타데이터로 추가 용어 사전 + LLM 태깅

    필요한 데이터¶

    항목 상세 비고
    LLM 구어체 HyDE 질문 생성 (문서당 2~3개) 일회성 배치 작업
    스토리지 포인트 수 증가 (현재 93,943 → +12,000~18,000 예상) Qdrant 스토리지
    리인덱싱 chunker → embedder → upserter 파이프라인 재실행 일회성

    예상 효과¶

    • 일회성 작업으로 영구적 효과 — 질의 시 추가 비용 없음
    • 구어체 사용자가 별도 전처리 없이도 관련 문서에 도달 가능

    비용/리스크¶

    • 포인트 수 증가에 따른 검색 성능/비용 고려
    • 현재 hyde 청크의 질문 스타일 점검 필요 — 이미 구어체가 포함되어 있다면 불필요
    • 리인덱싱 작업 시간

    7-5. 권장 적용 순서 및 파이프라인 설계¶

    단독 적용보다 조합이 효과적. 비용 대비 효과 순으로 단계별 적용을 권장한다.

    전체 파이프라인¶

    [사용자 질의 입력]
    ↓
① LLM 질의 정규화 + 변형 생성 (7-1 + 7-2 통합, LLM 1회 호출)
   - 정규화 질의 1개 + 변형 2개 생성
   - 비용: 경량 LLM 1회
    ↓
② 다중 질의 벡터 검색 (원본 + 정규화 + 변형 각 top-K)
   - 비용: 벡터 검색 3~4회 (ms 단위)
    ↓
③ 결과 합산 (RRF)
   - 비용: 무시 가능
    ↓
④ (선택) Reranker 재정렬
   - 비용: Cross-encoder 추론 (top-20~30 문서)
    ↓
[최종 top-K → LLM 답변 생성]

    단계별 적용 우선순위¶

    단계 방안 이유 추가 필요사항 예상 효과
    1단계 LLM 질의 정규화 + Multi-Query (7-1 + 7-2) 추가 데이터/모델 불필요, LLM만 있으면 즉시 적용 가능 프롬프트 설계, 용어 사전 작성, RRF 합산 로직 구현 키워드↔B 갭 0.31 → 0.05~0.10 축소. Overlap 대폭 개선 예상
    2단계 Reranker 도입 (7-3) 1단계로 확보한 후보군의 순서 최적화 한국어 Cross-Encoder 모델 선정, GPU 추론 인프라 구어체/전문용어 간 순위 편차 보완
    3단계 구어체 HyDE 보강 (7-4) 질의 전처리 없이도 구어체 매칭 개선 (영구적) 리인덱싱 파이프라인 실행, 기존 hyde 스타일 분석 질의 시 추가 비용 없는 근본적 개선

    1단계만으로도 Overlap 문제의 대부분을 해결할 수 있을 것으로 예상. 예: "집 여러 채 가진 사람 세금 불이익" → LLM이 "다주택자 세금 중과"로 정규화하면, 키워드↔질의B 갭이 사실상 0에 근접.
